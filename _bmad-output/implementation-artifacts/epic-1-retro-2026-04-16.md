## 做得好的

### 1. 基础架构一次到位
Story 1.1 建立的异常体系（AppError 层次）、CamelCase 中间件、TraceId 中间件、依赖注入模式，后续 5 个 Story 全部直接复用，零返工。

### 2. 安全意识贯穿始终
- bcrypt 固定版本 >=4.0,<4.1（避免 passlib 兼容问题）
- joserfc 替代被弃用的 python-jose
- 登录错误信息不泄露具体字段
- 密码从不出现在 API 响应中
- 操作幂等性（seed 脚本、成员绑定）

### 3. 渐进式复杂度
Stories 按依赖关系编排：auth → users → projects → members → RBAC，每层在前一层基础上构建，降低了单个 Story 的认知负担。

### 4. Service 层抽象清晰
路由层保持极薄（只做参数接收和响应返回），所有业务逻辑在 services/ 中实现，依赖注入（FastAPI Depends）贯穿全栈。

### 5. 技术选型明确
每个 Story 文档中明确指定了库版本和选型理由（如 joserfc 替代 python-jose），避免了选型纠结。

## 需要改进的

### 1. 测试是事后补写的（核心改进项）
**问题：** Epic 1 的所有代码先实现后补测试，未遵循 ATDD 先写验收测试再开发的流程。76 个 ATDD 测试是在全部代码实现完成后才生成的。

**影响：**
- 失去了 TDD 的"红-绿-重构"价值循环
- 测试可能基于实现而非需求，存在"通过即正确"的偏差
- 未能在开发阶段发现验收标准理解偏差

**行动项：** Epic 2 必须先运行 `bmad-testarch-atdd` 生成验收测试，再开始开发。

### 2. Sprint 状态追踪严重滞后
**问题：** sprint-status.yaml 显示 Story 1.1 in-progress、1.2~1.6 backlog，但实际上全部 6 个 Story 已完成。状态直到回顾前才一次性更新。

**影响：**
- 项目进度对外不透明
- 无法基于状态做决策
- 回顾时需要额外花时间逆向确认实际状态

**行动项：** 每个 Story 完成后立即更新 sprint-status.yaml。

### 3. 前后端 API URL 不对齐
**问题：** Story 1.2 文档中记录了前端 BASE_URL='/api/v1' 与后端路由 '/api' 不一致的问题。

**影响：** 前后端联调时可能出现 404。

**行动项：** Epic 2 开始前确认并统一 API 前缀。

### 4. tests/ 目录缺少 `__init__.py`
**问题：** 所有测试文件使用 `from tests.conftest import ...` 导入，但 tests/ 及子目录缺少 `__init__.py`，导致 pytest 运行时 `ModuleNotFoundError`。

**影响：** 已有的测试（包括 ATDD 测试）全部无法运行。

**修复状态：** 已在回顾当天修复（为所有测试目录添加 `__init__.py`）。

## 关键洞察

1. **"每完成一个 Task 立即验证"** — Story 1.3 提出的经验，避免问题积累到最后才发现
2. **三步事务必须原子化** — 创建项目 = 插入 project + branch + member，不可拆分
3. **依赖注入模式是扩展利器** — require_role / require_project_role 的工厂函数模式可复用到所有新端点
4. **数据库约束 + 应用层校验 双保险** — username 唯一约束在 DB 层和 service 层都有处理
