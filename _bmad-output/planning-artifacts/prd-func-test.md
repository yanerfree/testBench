# Product Requirements Document - testBench 功能场景测试模块

**Author:** Dreamer
**Date:** 2026-07-08

## Executive Summary

testBench 功能场景测试模块为测试工程师、测试 Lead 和开发工程师提供 AI 驱动的**功能测试场景用例**自动生成能力。用户提供需求材料（需求文档/PRD、用户故事、业务规则描述，可结合项目已有 API 定义与页面信息），AI 完成需求理解 → 场景建模（业务流程、状态机、角色权限矩阵）→ 生成结构化的功能测试场景用例，落入已有用例管理体系，经评审后进入测试计划执行并产出报告。

本模块解决的核心痛点：功能测试用例编写严重依赖人工经验且耗时（一个需求的场景用例设计需 1-2 天）；场景覆盖靠个人能力，跨模块业务流、状态流转、多角色交互场景容易遗漏；需求与用例之间没有追溯关系，需求变更后不知道哪些用例要更新；AI 生成的用例缺乏质量门禁，"可信的"和"待验证的"用例混在一起。

这是接口测试模块（已交付）验证过的"生成 → 评审 → 发布 → 执行 → 报告"AI 工作流向业务语义层的升级：输入从结构化接口 schema 变为弱结构需求材料，输出从单接口测试场景变为跨模块业务流程场景用例，复杂度显著更高，但平台底座（AI 配置体系、MCP 工具、SSE 流式交互、用例/计划/报告数据模型）全部可复用。

### What Makes This Special

**需求理解 + 场景建模，而非 schema 机械推导。** 接口测试的生成逻辑是"参数约束 → 测试维度推导"；功能场景测试的生成逻辑是"需求语义理解 → 业务流程/状态机/角色矩阵建模 → 场景用例推导"。AI 需要先输出中间产物（场景模型：业务流程图、状态转换表、角色权限矩阵、测试点清单），经用户确认后再展开为具体用例——两段式生成保证方向正确后才批量产出。

**需求-用例双向追溯。** 每条生成的用例回链到需求条目（需求点 ID），形成需求覆盖矩阵：哪些需求点已覆盖、覆盖了哪些维度（正向/异常/边界/权限）、哪些需求点零覆盖。需求变更时可定位受影响的用例集。

**评审门禁 + 质量评分。** AI 生成的用例带生成置信度与质量评分（步骤完整性、预期结果明确性、数据具体性），进入"待审核"状态，评审通过才可纳入测试计划——补上 AI 测试落地指南中列为 P0 的能力缺口。

**与既有资产协同。** 生成时读取项目已有用例做去重与互补（不重复生成已覆盖场景）、读取 API 定义补全接口级细节、读取环境配置生成可执行的数据准备步骤；生成的场景用例可后续联动 api_scenario/ui_scenario 与脚本自动生成线（tea_step 机制），从"手工执行的功能用例"升级为"可自动化的场景"。

**双通道生成。** 平台内可视化操作（上传/粘贴需求 → 场景模型确认 → 批量生成 → 评审）与 Claude Code CLI（读需求文档 + 源码，通过 MCP 工具生成同步到平台）两种方式，与接口测试模块保持一致的使用心智。

## Project Classification

- **项目类型：** Web 应用（SaaS/B2B 测试管理平台的功能模块）
- **领域：** DevTools / 测试工具
- **复杂度：** 高 — 弱结构输入的语义理解、两段式生成工作流、跨模块场景编排、需求追溯体系，均超出接口测试模块（中等复杂度）的范畴
- **项目上下文：** Brownfield — 在已有 testBench 平台上新增模块，深度复用用例管理、测试计划、报告、AI 配置、MCP、环境管理基础设施

## Success Criteria

### User Success

- **评审通过率 ≥ 80%**：AI 生成的场景用例，评审时不修改或仅轻微修改（改措辞不改结构）即通过审核的比例
- **场景模型一次确认率 ≥ 70%**：两段式生成中，AI 输出的场景模型（业务流程/状态机/角色矩阵/测试点清单）用户不做结构性调整直接确认的比例
- **用例可执行性 ≥ 90%**：抽检生成的用例，测试人员无需回看需求文档即可按步骤执行（步骤具体、数据明确、预期可判定）
- **需求零盲区**：每个提取的需求点在覆盖矩阵中有明确状态（已覆盖+维度 / 零覆盖），不存在"不知道测没测"的需求点
- **效率跃升**：单份需求文档从上传到用例入库 ≤ 30 分钟（AI 生成部分 ≤ 5 分钟，其余为人工确认与评审），替代当前 1-2 天的人工设计

### Business Success

- **场景遗漏可证伪**：试点模块中 AI 生成的场景至少发现 1 个人工用例库遗漏的有效场景（证明"补盲"价值）
- **AI 测试产品线统一**：与接口测试模块形成一致的"生成→评审→发布→执行→报告"心智，平台成为 AI 测试生成的完整解决方案
- **落地指南 P0 能力闭环**：用例审核状态 + 生成质量评分两项 P0 缺口在本模块交付，并可反向推广到接口测试模块

### Technical Success

- **场景模型生成 < 60 秒**，单需求点用例展开 < 30 秒，批量按需求点数线性增长
- **生成结果 100% 结构合法**：落库用例全部通过 Case 模型必填校验（标题、步骤非空、预期结果明确），无半成品数据
- **零新增 AI 配置面**：完全复用 `resolve_ai_config()` + `llm_client` 体系与 AIUsageLog 成本记录
- **生成失败可恢复**：LLM 调用失败/截断时已完成部分保留，支持断点续生成

### Measurable Outcomes

- 评审通过率 ≥ 80%（系统记录 `edited_after_generate` 与审核结果统计）
- 场景模型一次确认率 ≥ 70%
- 需求点覆盖矩阵完整率 100%
- 生成用例结构合法率 100%
- 单需求文档端到端处理时间 ≤ 30 分钟
- 双通道落地验证：试点期 Claude Code/MCP 通道被 ≥1 名用户真实使用并产出已审核用例（v1.2 补录，锚定「双通道生成」差异化）

## Product Scope

### MVP - Minimum Viable Product

**输入与需求理解：**
- 需求材料输入：粘贴文本 / 上传 Markdown 文档（其他格式 Growth）
- 辅助上下文：选择关联模块、勾选相关 API 定义、补充业务规则文本
- AI 提取需求点清单（每条带 ID），作为追溯锚点

**两段式生成：**
- 第一段：AI 输出场景模型 —— 业务流程步骤、状态转换表、角色权限矩阵、测试点清单（含维度标记：正向/异常/边界/权限/数据）
- 用户在平台上确认/编辑场景模型（增删测试点、修正流程）
- 第二段：按确认后的模型批量展开为场景用例，SSE 流式反馈

**用例落库与组织：**
- 生成结果写入已有用例管理（Case 模型），支持选择目标模块/文件夹，AI 可自动按需求结构归类
- 来源标记 `ai`，记录需求点追溯关系
- 与已有用例去重：生成前读取项目用例库，重复场景跳过并标注

**评审门禁：**
- 审核状态流转：待审核 → 已审核 / 已拒绝（拒绝需填理由，供 prompt 迭代）
- 质量评分：步骤完整性、预期明确性、数据具体性自动打分，低分用例高亮提示重点评审
- 批量审核操作

**覆盖矩阵：**
- 需求点 × 测试维度矩阵视图，展示覆盖状态与用例链接、零覆盖需求点告警

**双通道：**
- 平台内向导式界面（输入 → 模型确认 → 生成 → 评审）
- MCP 工具扩展（Claude Code 读本地需求文档/源码生成，同步到平台，复用 API Key 认证）

**执行与报告（复用既有能力）：**
- 已审核用例可加入测试计划，走已有手动计划执行录入
- 报告类型启用预留的「功能场景测试」

**生成质量纵深（v1.2 补录，对应 FR51-FR59）：**
- 输入预处理与消毒、封闭维度枚举、结构化输出校验与受限重试、静态质检与覆盖补漏、落库前 AI 自评回炉、生成档案、已审核样板基线、可自动化写作契约

**用户体验基线（v1.2 补录，对应 FR61-FR69）：**
- 向导单页流与任务断点恢复、双栏产出面板、进度回放、审核键盘流、用例全景与深链、矩阵就地补生成、成本预估、真实空态与失败原因必现

### Growth Features (Post-MVP)

- AI 优化：对已有用例提建议 → AI 给方案 → 确认执行（复用接口测试模式）
- 需求变更影响分析：重新上传需求 → diff 需求点 → 定位受影响用例集并提示更新
- 自动化联动（脚本质量线，v1.1 具体化）：场景用例按可自动化写作契约（FR59）一键生成 api_scenario / ui_scenario 与脚本草稿；脚本走 draft → execute → finalize/discard 转正状态机——**真实执行通过才成为正式资产**（≤3 轮修复，失败不保存）；前置造数经 setupRef 绑定已验证的 Setup 代码；衔接脚本自动生成线（tea_step 机制）
- llm-mock 服务（OpenAI/Anthropic 兼容双端点，可注入延迟/错误/固定响应），把「提取→建模→展开→自评→审核」全链路写成 E2E 用例进 CI，覆盖 AI 超时/格式错误降级分支
- 更多输入格式：Word / PDF / 原型图截图（多模态）/ 链接抓取
- 生成质量评分推广到接口测试模块

### Vision (Future)

- 探索式测试联动：结合 Exploratory 模块的页面探索结果反哺场景建模
- 领域测试模式库：沉淀行业/项目级场景模板与业务规则知识库，生成质量随使用持续提升
- 全自动回归编排：需求变更 → 影响分析 → 用例更新 → 自动化脚本再生成 → 回归执行的闭环

## User Journeys

### 旅程一：测试工程师 — 平台内从需求到用例（主力场景）

**小陈**，测试工程师，不写代码。周一拿到「订单退款」新需求的 PRD，按老办法要写两天用例，还总担心漏场景。

1. 打开功能场景测试入口，粘贴需求文档（Markdown），选择关联模块「订单」，勾选项目里已录入的退款相关 API，补充一条业务规则："部分退款后订单状态保持已发货"
2. AI 先做需求质量检测：提示 1 条歧义描述（含原文引用与修改建议），小陈补了句业务说明；健康分 82 未触发低分提示。随后 AI 提取出 12 个需求点（每条带 ID 和原文引用），小陈扫一遍确认没有漏读
3. AI 输出场景模型：退款业务流程步骤、订单状态转换表（已支付→退款中→已退款/退款失败）、角色权限矩阵（买家/客服/财务）、31 个测试点（标注维度：正向/异常/边界/权限/数据）
4. 小陈发现模型里漏了"审批驳回后重新提交"的流转，在模型编辑界面补上这条状态转换和 2 个测试点，确认模型
5. 第二段生成启动，SSE 流式展示，42 条场景用例逐条落库为「待审核」，自动归入「订单/退款」文件夹
6. 按质量评分排序评审：高分用例快速通过，3 条低分用例预期结果含糊，填理由拒绝
7. 覆盖矩阵显示 12 个需求点中 11 个已覆盖，1 个零覆盖（性能类需求）——标注说明转 NFR 处理
8. 39 条已审核用例批量加入测试计划，按步骤手动执行，报告类型为「功能场景测试」

**关键时刻**：两天的活半小时干完，而且覆盖矩阵让她第一次能回答"这个需求测全了吗"。

### 旅程二：测试工程师 — 生成中断与去重（边缘场景）

**小陈** 处理一份 40 页的大需求文档。

1. 需求点提取出 35 条，场景模型确认后开始批量生成
2. 生成到第 25 条时 LLM 调用超时中断——已生成的 25 条完好保留在「待审核」，界面提示失败位置和原因
3. 点「继续生成」，从第 26 条断点续生成，不重复消耗 token
4. 完成后结果摘要显示：新增 52 条、跳过 6 条（与已有用例重复，列出重复对应关系）、失败重试 1 次
5. 小陈点开被跳过的记录，确认确实是上个迭代已覆盖的场景，放心关闭

### 旅程三：测试 Lead — 评审治理与覆盖督导

**老周**，测试 Lead，负责把关 AI 生成质量。

1. 打开待审核队列，按质量评分升序排列，先看最可疑的
2. 批量选中评分 ≥ 85 的 20 条快速通过；对低分用例逐条审，发现共性问题：测试数据用了"某用户"这类模糊表述——点开其中一条的「生成档案」时间线，确认该问题从上个 prompt 版本起就反复出现
3. 拒绝时填写理由，系统汇总拒绝理由分布，老周把"测试数据必须具体"反馈给平台管理员用于 Skill prompt 迭代
4. 查看覆盖矩阵：「支付退款」需求点只有正向维度覆盖——发起补充生成，指定该需求点补异常+边界维度
5. 月底用统计页回答老板："本月 AI 生成 300 条，评审通过率 84%，发现人工遗漏场景 7 个"

### 旅程四：开发工程师 — Claude Code 通道生成

**小李**，全栈工程师，习惯在 IDE 里干活，本地仓库有需求文档和源码。

1. testBench MCP 工具页面已配置过（接口测试时接入的，无需重复配置）
2. 在 Claude Code 输入："根据 docs/prd-refund.md 和退款模块源码，生成功能测试场景用例"
3. Claude Code 读需求文档 + 源码（真实的状态机实现、错误码定义），通过 MCP 工具提交场景模型与用例到平台
4. 源码信息让用例更精准：错误提示文案、边界值与代码实际校验一致
5. 平台上测试同事收到待审核用例，评审流程与平台内生成完全一致

### 旅程五：管理者 — 查看功能场景测试报告

**张主管**，只读权限。

1. 打开测试报告页面，筛选类型「功能场景测试」
2. 查看退款需求的执行报告：42 条用例，通过 39、失败 2、阻塞 1
3. 下钻失败用例查看执行录入的实际结果与备注
4. 结合覆盖矩阵回答"退款功能可以上线吗"

## Domain-Specific Requirements

### AI 生成质量与幻觉控制（本域核心风险）

- **需求点必须带原文引用**：每个提取的需求点附带需求文档原文片段与位置，用户可一键回查比对，杜绝 AI 编造需求
- **用例标注测试设计方法**：每条生成用例标注所用方法（场景法/等价类/边界值/状态迁移/判定表/错误推测），生成过程可解释、可审计
- **禁止臆造事实**：接口路径、错误文案、字段名等事实性内容只能来自输入材料（需求文档/API 定义/源码），输入中没有的必须显式标注"待确认"而非编造
- **结构化输出强校验**：LLM 输出经 JSON Schema 校验，不合法自动重试；半成品不落库

### 数据安全与隐私

- 需求文档可能包含未发布的商业信息：发送给 LLM 的内容范围对用户可见（生成前可预览上下文组成）
- 复用项目级 AI 配置：企业可将本模块指向私有化部署的 LLM，数据不出内网
- AIUsageLog 记录调用元数据（token/耗时/模型），不落盘需求文档原文之外的冗余副本；日志中脱敏

### 成本与容量约束

- 大文档处理有 token 预算保护：超长需求文档分块提取需求点，单次生成任务有 token 上限与用户可见的成本预估
- 两段式生成天然控本：场景模型（小成本）确认后才展开批量生成（大成本），方向错误不浪费

### 与既有平台域的一致性约束

- **用例数据兼容**：生成用例完全符合平台主 PRD（FR-CASE-001）的用例数据结构（TC 编号规则、模块归属、步骤格式），不破坏 tea-cases.json 导入链路与 `tea_id` 体系（AI 生成用例 `source=ai`，与 `imported`/`manual` 并列）
- **分支隔离**：生成用例挂在当前分支下，遵循既有分支数据分类规则（用例随分支复制）
- **RBAC**：复用平台 5 级角色 —— 生成与评审需编辑权限，审核通过操作限 Lead/管理员角色（可配置），游客只读
- **操作审计**：生成/审核/拒绝/删除操作写入既有操作日志体系
- **MCP 架构沿用**：复用接口测试模块已解耦的真实 MCP Server 与 API Key 认证，新增工具注册即可

### 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| AI 幻觉生成不存在的需求/接口 | 用例不可信，评审负担反增 | 需求点原文引用 + 事实性内容只来自输入 + 待确认标注 |
| 需求文档写得太烂，AI 无从理解 | 生成质量差，用户弃用 | 需求点提取阶段即提示"需求描述不完整"清单，引导补充业务规则；质量差的输入生成前拦截 |
| 生成用例与人工用例风格割裂 | 用例库混乱 | 生成遵循项目已有用例的步骤粒度与措辞风格（few-shot 采样已有优质用例） |
| 大批量生成淹没评审 | 门禁形同虚设 | 质量评分排序 + 批量审核 + 单次生成量上限 |
| LLM 服务不稳定 | 生成中断体验差 | 断点续生成 + 部分结果保留 + 失败原因透出 |

## Innovation & Novel Patterns

### Detected Innovation Areas

本模块属于**组合式创新**（成熟技术的新组合），而非单点技术突破：

1. **场景模型作为可确认的中间产物**：业内 AI 用例生成工具（含各类 ChatGPT 套壳方案）普遍是"需求进、用例出"的黑盒一段式。本模块把"AI 对需求的理解"（业务流程/状态机/角色矩阵/测试点清单）显式化为可编辑、可确认的中间产物——用户纠正的是"理解"，而不是逐条修 500 行生成结果。这是对"AI 生成不可信"痛点的结构性回应。
2. **需求-用例追溯矩阵内建于测试平台**：需求覆盖追溯传统上属于 ALM 工具（如 Polarion）的重量级能力；将其轻量化为"AI 提取需求点 + 生成时自动建链"，在测试管理平台中零额外录入成本地获得覆盖矩阵。
3. **评审门禁与质量评分作为 AI 生成的一等公民**：拒绝理由结构化收集 → prompt 迭代的反馈回路，把"AI 质量改进"从工程师的手工调 prompt 变成产品化机制。

### Market Context & Competitive Landscape

- 商业工具（mabl、Testim、Functionize 等）聚焦 UI 自动化脚本 AI 生成，功能测试**用例设计**层的 AI 产品化程度低
- 国产测试平台（MeterSphere 等）的 AI 用例生成多为一段式黑盒，无需求追溯与质量门禁
- testBench 差异化：接口测试（已交付）+ 功能场景（本模块）+ 脚本生成线（进行中）构成生成-评审-执行-报告全链路，且双通道（平台 + Claude Code/MCP）

### Validation Approach

- 试点验证：选 1 个真实需求（建议用 testBench 自身的新需求文档）走全链路，验收"评审通过率 ≥ 80%、发现 ≥ 1 个人工遗漏场景"
- 场景模型价值验证：对比"带模型确认"与"直接生成"两种模式的评审通过率差异（内部 A/B）
- 数据埋点先行：`edited_after_generate`、审核结果、拒绝理由自 MVP 起记录

### Risk Mitigation

- 若场景模型确认环节被用户认为繁琐（一次确认率高但感知负担重）→ 提供"跳过确认直接生成"快捷模式，模型仍生成但默认折叠
- 若需求点提取质量不稳定 → 降级路径：用户手工框选需求段落代替 AI 提取，追溯链路不受影响
- 若质量评分与人工判断偏差大 → 评分仅作排序参考不作硬门禁，保留人工审核为唯一权威

## Web App 技术需求

### 架构概述

- **前端**：React 19 SPA（已有），Ant Design，Vite；本模块新增向导式生成界面与覆盖矩阵视图
- **后端**：FastAPI + PostgreSQL + SQLAlchemy async（已有）
- **AI 调用**：复用 `resolve_ai_config()`（项目级 → 系统级 → .env 回退）+ `llm_client`（openai_compatible / anthropic），SSE 流式
- **MCP Server**：复用 FastMCP `/mcp` 路由与 API Key 认证，注册功能场景测试工具集
- **实时性**：SSE 流式（生成进度、逐条落库反馈），复用前端 `api.stream()` 封装；无 WebSocket 需求
- **SEO**：内部 B2B 平台，无需求
- **浏览器矩阵**：Chrome / Edge / Firefox 最新版（与平台一致）
- **响应式**：桌面优先（1280px+），与平台既有页面一致，不做移动端适配
- **可访问性**：遵循 Ant Design 默认无障碍能力，不做额外 WCAG 认证

### 新增技术组件

| 组件 | 说明 |
|------|------|
| 生成任务编排 | 两段式生成的任务状态机（需求提取 → 模型待确认 → 生成中 → 完成/部分失败），支持断点续生成，任务状态持久化 |
| 需求点与追溯数据模型 | 需求点（含原文引用）、场景模型、需求点-用例追溯关系三类新实体 |
| 质量评分器 | 规则+LLM 混合打分（步骤完整性/预期明确性/数据具体性），生成后自动执行 |
| 用例审核工作流 | Case 增加审核状态（待审核/已审核/已拒绝）与拒绝理由，含批量审核 |
| 去重比对器 | 生成前语义比对已有用例（标题+步骤摘要），重复项跳过并记录对应关系 |
| 覆盖矩阵视图 | 需求点 × 维度矩阵前端组件 + 聚合查询 API |
| MCP 工具扩展 | 提交需求材料/场景模型/用例、查询覆盖矩阵等工具 |
| Skill 模板 | 平台内 Skill：需求点提取、场景建模、用例展开、质量评分四个 prompt 模板（`backend/app/skills/preset/` 体系） |

### 与已有系统的集成点

- **用例管理**：生成结果落 `Case` 模型（`source=ai`），归入 `CaseFolder` 树；不改变既有导入/导出/同步链路
- **测试计划/报告**：已审核用例进入既有计划执行流，报告启用 `report_type=功能场景测试`
- **AI 配置体系**：零新增配置面，AIUsageLog 记录成本
- **API 接口管理**：读取 API 节点定义作为生成辅助上下文
- **审计日志**：生成/审核操作接入既有 logs 体系
- **既有生成链路关系**：`tb-case-generate`（基于接口定义的单点用例生成）保留；本模块是独立的需求驱动场景生成链路，二者共存，入口分开、命名区分

### Implementation Considerations

- 两段式任务的中间态（场景模型）需要持久化，用户可能隔天回来继续——任务不能只存在内存/SSE 会话里
- 大文档分块提取需求点时，跨块的需求关联（如"见 3.2 节"）需要合并处理策略
- 质量评分应异步执行，不阻塞用例落库
- MCP 通道与平台通道生成的数据结构完全一致，评审侧无感知差异

## Project Scoping & Phased Development

### MVP Strategy & Philosophy

**MVP 方针：** 问题验证型 MVP —— 优先验证"AI 生成的功能场景用例质量是否可信、评审负担是否可接受"这一核心假设，而非功能广度。生成质量不达标，其余一切（矩阵、通道、联动）都没有意义。

**资源假设：** 沿用接口测试模块的单人全栈 + AI 结对开发模式；Skill prompt 打磨预留专门时间（预计占开发量 30%，高于接口测试模块，这是复杂度差异的主要体现）。

**验证路径：** 用 testBench 自身的真实需求文档（如本 PRD 对应的功能）做全链路试点，吃自己的狗粮。

### MVP Feature Set (Phase 1)

**支撑的核心旅程：** 旅程一（平台内生成）、旅程二（中断与去重）、旅程三（评审治理）、旅程五（报告查看）

**Must-Have（缺一 MVP 即不成立）：**
1. 需求输入（粘贴/上传 Markdown）+ 需求点提取（带原文引用）
2. 两段式生成：场景模型确认 → 批量展开用例（SSE 流式、断点续生成）
3. 用例落库（Case 模型，source=ai，需求点追溯关系）
4. 审核工作流（待审核/已审核/已拒绝 + 拒绝理由 + 批量审核）
5. 质量评分（排序参考，非硬门禁）
6. 覆盖矩阵（需求点 × 维度）
7. 计划/报告集成（手动执行，report_type=功能场景测试）
8. 生成质量纵深基线（FR51-FR59：预处理、枚举约束、校验重试、静态质检、自评回炉、生成档案、样板基线、写作契约）（v1.2 补录）
9. 用户体验基线（FR61-FR69）（v1.2 补录）

**MVP 内可裁剪（进度压力下的降级顺序，先裁前面的）：**
1. MCP 通道（旅程四）→ 后置到 Phase 1.5，平台内通道先行验证质量
2. 已有用例去重 → 降级为"同一生成任务内去重"
3. 自动归类 → 降级为用户手选目标文件夹
4. 质量评分 → 降级为规则打分（不调 LLM）

### Post-MVP Features

**Phase 2 (Growth)：** AI 优化（建议→方案→确认）、需求变更影响分析、自动化联动（可自动化契约衔接 api_scenario/ui_scenario + 脚本 draft→execute→finalize 转正状态机 + 脚本生成线）、llm-mock 驱动的 AI 链路 E2E 进 CI、prompt 试跑台、更多输入格式（Word/PDF/图片）、评分推广到接口测试、生成质量时序仪表盘（每日快照：生成量/通过率/拒绝率/修改率）

**Phase 3 (Vision)：** 探索式测试联动、领域测试模式库、需求变更→回归执行全自动闭环

### Risk Mitigation Strategy

- **技术风险（最大）：需求理解与场景建模质量不稳定** → 两段式生成把风险截断在模型确认环节；试点先行打磨 prompt；提供手工框选需求段落的降级路径
- **体验风险：确认环节增加操作负担** → 提供"跳过确认"快捷模式；模型编辑交互按最小可用设计（表格增删改，不做图形化流程编辑器）
- **市场/用户风险：评审通过率低于预期导致弃用** → 埋点自 MVP 起收集，拒绝理由结构化回流 prompt 迭代；设立 80% 通过率的止损评估点
- **资源风险：** 按上方裁剪顺序收缩范围；MCP 通道、去重、自动归类均可后置而不动摇核心假设验证

## Functional Requirements

### 可配置阈值默认值表（v1.2）

以下阈值均为系统级出厂默认，项目级可配置；FR 文本中引用此表，不再各自内联数值。

| 参数 | 默认值 | 关联 FR |
|------|--------|--------|
| 需求健康分提示阈值 | < 70 分触发低分确认提示 | FR5 |
| 超长文档判定 | 需求材料 > 16k 字符（约 10 页）触发分块，块大小 16k 字符 | FR6、FR51 |
| 输入长度上限 | 单次任务需求材料 ≤ 200k 字符 | FR51 |
| P0 占比上限 | > 40% 时超额 P0 自动降为 P1 并记 warning | FR54 |
| 模糊断言红线词表 | 内置基础词表（操作成功/显示正常/无报错/符合预期等），项目级可增删 | FR54 |
| AI 自评通过阈值 | ≥ 75 分；回炉上限 3 轮 | FR56 |
| 结构化输出重试上限 | 2 次 | FR53 |
| 拒绝理由注入条数 | 最近 5 条（可配 1-20） | FR26 |
| 去重判定 | 标题相似度 ≥ 0.7 且步骤关键动作重叠 ≥ 50% 判疑似重复，跳过并标注 | FR18 |
| 单任务用例数上限 | 200 条 | FR38 |

### 1. 需求输入与需求点提取

- **FR1**: 用户可以粘贴文本或上传 Markdown 文件作为需求材料发起生成任务
- **FR2**: 用户可以为生成任务补充辅助上下文：关联模块、项目已有 API 定义、业务规则文本
- **FR3**: AI 自动从需求材料提取需求点清单，每条需求点带唯一 ID 与原文引用（片段+位置），可一键回查
- **FR4**: 用户可以编辑需求点清单（增/删/改/合并），也可以手工框选需求段落新建需求点（AI 提取的降级路径）
- **FR5**: AI 对需求材料做三维质量检测（逻辑矛盾/边界缺失/歧义描述），每条问题含分类、严重度、原文引用、修改建议；健康分由系统按严重度加权计算（LLM 只出问题清单不打分）；健康分低于阈值（见阈值表，默认 <70）时提示用户确认后**仍可继续生成**（软门禁，不做阻断状态机）；用户忽略/删除问题项记录操作人（v1.2 修订）
- **FR6**: 超过阈值表定义的超长需求文档自动分块提取需求点（默认 >16k 字符触发），跨块结果自动合并去重（v1.2 修订）

### 2. 场景建模（两段式第一段）

- **FR7**: AI 基于需求点生成场景模型，包含业务流程步骤、状态转换表、角色权限矩阵、测试点清单四类结构
- **FR8**: 每个测试点标注测试维度（正向/异常/边界/权限/数据）与关联需求点
- **FR9**: 用户可以编辑场景模型：增删改测试点、修正流程步骤、状态转换与角色定义
- **FR10**: 用户确认场景模型后才进入用例展开；提供"跳过确认直接生成"快捷模式（模型仍生成并保留）
- **FR11**: 场景模型持久化，用户可以中途离开后继续处理（任务不因会话结束丢失）
- **FR12**: 用户可以针对指定需求点或指定维度发起增量补充生成，不重做全量

### 3. 用例生成与落库

- **FR13**: AI 按确认后的场景模型批量展开场景用例，按测试点粒度逐条生成与跟踪，单条失败不阻塞后续并记录失败环节与可读原因；SSE 流式反馈进度与逐条落库结果（v1.2 修订：任务结构细节移至架构设计）
- **FR14**: 每条生成用例包含：标题、优先级、前置条件、有序步骤（操作+单步预期）、整体预期结果、具体测试数据
- **FR15**: 每条用例标注所用测试设计方法（场景法/等价类/边界值/状态迁移/判定表/错误推测）与关联需求点、测试点
- **FR16**: 事实性内容（接口路径、错误文案、字段名）只来自输入材料；输入中不存在的信息标注"待确认"，禁止编造
- **FR17**: 生成用例写入既有用例管理（Case 模型，source=ai），用户可选目标模块/文件夹，或由 AI 按需求结构自动归类
- **FR18**: 生成前与项目已有用例语义比对去重，重复场景跳过并展示与已有用例的对应关系
- **FR19**: 生成完成展示结果摘要：新增/跳过（去重）/失败计数与明细
- **FR20**: 生成用例的步骤粒度与措辞风格对齐项目已有「已审核」状态用例（选取基线同 FR58）（v1.2 修订）

### 4. 评审与质量门禁

- **FR21**: AI 生成用例初始为「待审核」状态，仅「已审核」用例可加入测试计划
- **FR22**: 审核人可以通过或拒绝用例，拒绝必须填写结构化理由（问题分类枚举 + 文本说明）；并发审核冲突被检测并提示刷新，不产生静默覆盖（v1.2 修订）
- **FR23**: 系统对每条生成用例自动质量评分，评分 = 零 token 规则静态校验（可解释）+ AI 四维加权自评（完整性 30/准确性 25/有效性 25/可执行性 20）合成（两分独立展示，总分默认 50/50 加权），作为评审排序参考而非硬门禁（v1.2 修订）
- **FR24**: 待审核队列支持按评分/需求点/维度/生成任务筛选与排序，支持批量通过、批量拒绝
- **FR25**: 审核人可以在审核前直接编辑用例内容，编辑行为被系统记录，支撑「生成后编辑率」度量（v1.2 修订）
- **FR26**: 拒绝理由结构化落库并汇总统计；同需求/同模块的后续生成自动把最近 N 条（默认 5，见阈值表）拒绝理由注入 prompt 的「避免以下问题」区块，注入前经安全隔离处理防止用户文本被当作指令执行；同时驱动 Skill prompt 版本迭代——先定消费点再定生产端（v1.2 修订）
- **FR27**: 手动创建与导入来源的用例不进入审核门禁，现有流程不受影响
- **FR28**: 审核通过/拒绝操作记录操作人与时间，可追溯

### 5. 覆盖矩阵与需求追溯

- **FR29**: 系统提供需求点 × 测试维度覆盖矩阵视图，展示每格覆盖状态与关联用例链接
- **FR30**: 零覆盖需求点显式告警，用户可从矩阵单元格直接发起补充生成
- **FR31**: 双向追溯：用例详情可查看关联需求点与原文引用；需求点可列出全部关联用例
- **FR32**: 覆盖矩阵可按生成任务/模块/文件夹过滤
- **FR33**: 需求点可标注「不适用/转 NFR」状态并注明原因，矩阵中区分显示，不计入零覆盖告警

### 6. 生成任务管理

- **FR34**: 生成任务全生命周期状态可见：需求提取 → 模型待确认 → 生成中 → 已完成/部分失败
- **FR35**: 生成中断（LLM 失败/超时/截断）时已完成部分保留，用户可从断点继续生成

## Non-Functional Requirements

### 性能

- **NFR1**: 需求点提取（≤10 页文档）与场景模型生成各自 < 60 秒；超长文档分块时进度可见
- **NFR2**: 用例批量展开按需求点粒度流式落库，单需求点展开 < 30 秒，任务整体耗时随需求点数线性增长
- **NFR3**: 页面加载与非 AI 生成类用户操作（列表/筛选/排序/表单提交/矩阵查看）响应 < 1 秒；覆盖矩阵在 200 需求点 × 6 维度规模下渲染 < 2 秒（v1.2 修订）
- **NFR4**: SSE 进度事件从产生到前端展示延迟 < 2 秒

### 安全

- **NFR5**: 需求材料仅发送至项目 AI 配置指向的 LLM 端点，发送内容范围对用户可见（上下文预览）
- **NFR6**: MCP API Key 沿用 SHA256 哈希存储与 Bearer Token 传输机制
- **NFR7**: 操作日志与 AIUsageLog 不记录密钥明文，涉及账号密码的用例数据在日志中脱敏

### 可靠性

- **NFR8**: 生成中断时已完成用例 100% 保留；断点续生成幂等，不产生重复用例
- **NFR9**: LLM 输出 JSON 解析失败自动重试至少 1 次；重试仍失败则任务标记"部分失败"并透出可读原因，不静默丢弃
- **NFR10**: 质量评分失败不阻塞用例落库（降级为"未评分"状态，可手动触发重评）
- **NFR11**: 并发生成任务下用例编号（TC-{MODULE}-{seq5}）无冲突，模块内单调递增

### 容量与可扩展性

- **NFR12**: 单次生成任务支持 ≤ 100 需求点、≤ 200 条用例；用例库 1 万条规模下待审核队列与覆盖矩阵查询 < 500ms
- **NFR13**: 不同项目/用户的生成任务可并行执行，互不阻塞（沿用平台异步任务与信号量模式）

### 集成与兼容

- **NFR14**: MCP Server 兼容 MCP 2025-03-26 规范（StreamableHTTP，沿用既有实现）
- **NFR15**: 对既有用例管理零破坏：tea-cases.json 导入、Excel 导出、测试计划、报告等既有流程回归通过
- **NFR16**: 支持 Chrome / Edge / Firefox 最新版（与平台一致）
- **NFR17**: 服务重启后进行中的生成任务被自动识别并恢复或标记失败（不留永久「生成中」僵尸态）；后台任务不因资源回收静默丢失，超时任务被自动终止并透出原因（v1.2 修订）
- **NFR18**: 双通道一致性：MCP 工具必须复用平台 service 层的校验、状态机与审计逻辑，禁止在 MCP 工具层旁路直写数据库（防审核门禁被 Claude Code 通道绕过）
- **NFR19**: Skill/prompt 资产以版本化存储为唯一事实源（版本号、变更说明、diff、回滚），平台通道与 Claude Code 通道消费同一版本，文件仅作初始种子——多来源漂移视为缺陷（v1.2 新增，由原 FR60 降级而来）

### 关键分歧裁决

**需求门禁：硬阻断（themisai 现存）vs 整体退役（Aemeath）** —— 裁决为中间路线：保留检测能力（它的 prompt/评分/剥离资产成熟），去掉阻断语义（它的状态机耦合被证死）。检测产出问题清单与健康分，低分需用户显式确认，永不设 gate 状态。

**空白确认**：「需求点原文引用锚定」在两个对标项目中均无实现（themisai 的 location 仅自由文本描述，Aemeath 只有文档级 FK）——本模块此项为自研首创，实现策略：提取时同步输出原文精确引用（exact quote），入库时字符串匹配校验引用真实存在，匹配失败降级标注，不做事后对齐。工作量估算按无对标可抄计。
