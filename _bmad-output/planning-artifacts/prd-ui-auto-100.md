# PRD — UI/接口自动化：单用例 100% 可用 + 自动化数据管理 + 场景变量

## 1. 背景与目标（Context / Goal）

testBench 的 UI 脚本生成已从"不可用"做到"CLI 引擎(真 claude CLI 直驱 Playwright MCP)可跑通简单用例"，但**单个用例的生成+执行还做不到 100% 可用**：需要"带鉴权 API 造数"的用例（如"禁用运行中的服务"）常失败（401、登录选择器、前置数据缺失）。

**硬目标（本 PRD 唯一衡量标准）**：**单个功能用例 → AI 生成一次 → 生成的脚本不依赖 AI、可反复执行、稳定通过（≥ 目标成功率）**。达不成不做批量（批量为后续 epic）。

支撑手段（用户提出、且与参考项目 ThemisAI/Aemeath 已验证模型一致）：**项目级自动化数据管理** + **场景变量（UI/接口共用）**。

## 2. 用户与场景（Personas）
- **测试工程师**：写功能用例 → 一键生成自动化脚本 → 反复回归执行，减少手工。
- **自动化维护者**：维护项目级全局数据（账号/共享资源），调试脚本。

## 3. 现状与痛点
- 生成脚本对"业务数据前置"处理不当：要么假设环境已有数据（脆），要么盲目造数但无鉴权(401)、无清理→脏数据。
- 全局登录每个脚本重复做，慢。
- UI 与接口测试各自维护数据，不共享。
- 生成的脚本硬编码值、无唯一性，调试难、批量跑互相污染。

## 4. 核心设计（最优方案）

### 4.1 项目级自动化数据（Automation Data Registry）
项目下新增"自动化数据"配置，分两类：
- **全局凭证（多角色）**：账号/密码（如 admin / tenant），登录一次→回填 token（缓存、按 TTL/失效刷新），供所有脚本复用，避免频繁登录。
- **全局共享资源**：具名的长期共享数据（如某个默认上游/负载/服务），带"存在性检查"定义（查询 API/条件）与可选"创建定义"（创建 API + 参数）。
- **预检机制**：跑自动化前，对用例声明依赖的全局数据做存在性检查 → **缺则按定义补建，或提示用户确认** → 齐则开跑。全局数据**建一次长期保留，绝不被测试删除**。

### 4.2 场景变量（Scenario Variables，UI/接口共用）
- 每个测试场景/用例挂一组**场景变量**；该场景用到的数据都提为场景变量。
- **UI 脚本与接口测试共用同一份场景变量**（同一数据定义，两种执行器都读）。
- **数据唯一性**：变量值可带随机后缀（如 `svc_${runId}_${rand}`），使每次运行数据唯一、可追溯（能看清本脚本造了哪些数据）、批量互不污染。
- **全局变量引用**：脚本以 `${var}` 形式引用全局数据/凭证，不硬编码。
- **步骤链不入变量**：场景内"上一步提取、下一步使用"的中间值走步骤内传递（extract→use），不提为场景变量。

### 4.3 全局 vs 场景 判定与前后置
- **全局数据**：查存在 / 变量引用；**只用不删**；缺则补建或让用户确认。
- **场景数据**：脚本**自建（唯一命名）+ cleanup 自删**；API 造数/清理**必须带鉴权**（回填 token → Authorization: Bearer / 或浏览器内 fetch credentials）。
- 生成时按用例前置措辞 + 项目自动化数据配置判定归类。

## 5. 功能需求（FR）— 单用例 100% 可用
- FR1 项目自动化数据 CRUD：全局凭证(多角色)、共享资源(存在性/创建定义)的维护界面 + 存储。
- FR2 token 回填：登录一次缓存 token，脚本执行时注入，失效自动刷新。
- FR3 场景变量：用例级场景变量的定义/编辑/引用，UI 与接口共用；随机后缀唯一化。
- FR4 生成期：区分全局/场景，场景级自建自删+唯一命名+带鉴权，全局级查存在/变量引用；禁硬编码绝对 URL/值。
- FR5 执行期预检：跑前检查全局数据存在→缺则补/确认；注入变量+凭证；不依赖 AI，可反复执行。
- FR6 生成脚本有效性门禁：verify 通过才转正 active；自愈≤3 轮；失败如实报（不假通过）。

## 6. 非目标 / 后续（Out of Scope — 低优）
- **C 批量执行**（后续 epic）：两入口（用例页选→自动建计划→报告；测试计划页建计划执行），选择时**区分执行 UI 脚本 vs 接口脚本**。仅在单用例 100% 达成后启动。

## 7. 验收标准（单用例 100% 的定义）
- 对 stoa 上一组代表性用例（含展示类、登录+创建类、状态变更类=需前置数据类），**AI 生成一次 → 脚本反复执行（不依赖 AI）稳定通过**，目标通过率（如 ≥95%，具体阈值待定）。
- 场景数据自建自删、不留脏数据；全局数据只用不删；无假通过（测错应用/永真断言）。
- UI 与接口共用场景变量落地可查。

## 8. Epics（初步）
- Epic 1: 项目级自动化数据（凭证+共享资源+预检+token回填）
- Epic 2: 场景变量模型（UI/接口共用+唯一化+引用）
- Epic 3: 生成期全局/场景前后置 + 鉴权造数（打通硬用例）
- Epic 4: 执行期预检+变量注入+有效性门禁（单用例 100%）
- Epic 5(低优): 批量执行 + UI/接口脚本区分

## 9. 已决策（Dreamer 确认 2026-07-20）
1. **全局共享资源缺失** → **提示用户确认后补建**（提供一键补建按钮，但不自动建，避免污染共享环境）。
2. **场景变量存储** → **新表 `scenario_variables`**（挂用例/场景，UI+接口共用；与跨用例的 environment variables 分开）。
3. **单用例 100% 验收** → 以"代表性用例集（展示 / 登录+创建 / 状态变更 三类各若干）全部通过"为准；单次生成成功率目标高 + 自愈兜到接近 100%（阈值实施时定）。
4. **token 缓存** → Redis，按角色缓存 + TTL + 失效刷新。

## 10. 下一步（BMAD）
架构设计（scenario_variables 表 + 项目自动化数据模型 + 生成/执行集成）→ Epic/Story 拆分 → 逐 Story 开发 + 自测。Epic 1/2（数据模型）为基础，Epic 3/4（生成/执行打通 100%）为目标，Epic 5（批量）低优。
