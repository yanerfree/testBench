# UX 设计规范 — 测试管理平台

**作者：** Dreamer
**日期：** 2026-04-13

---

## Executive Summary

### 项目愿景

测试管理平台是一个以「自动化为主、手动为辅」的轻量级测试管理工具，深度集成 BMAD TEA 框架的脚本输出。不做用例编辑器，专注做好"管理 + 执行 + 看结果"三件事，让测试团队从碎片化的脚本和散落的手动用例中解放出来，通过统一平台完成日常测试工作的闭环。

**差异化定位：** 与传统测试管理平台（TestRail / ZenTao）不同，本平台原生支持「自动化脚本 ↔ 结构化用例」双向集成，支持自动+手动混合执行，是 TEA 框架在企业内部的"工作台"。

### 目标用户

5 类角色，权限分级：

| 角色 | 使用频次 | 核心诉求 |
|------|---------|---------|
| **测试人员**（高频）| 每日多次 | 快速找到用例、批量勾选执行、看清楚失败原因 |
| **测试 Lead**（中频）| 每日 1-2 次 | 协调团队分发用例、查看整体进度、导出周报 |
| **项目管理员**（低频）| 周级 | 创建项目、配置仓库、管理成员 |
| **开发**（偶发）| 失败时进入 | 快速定位与自己模块相关的失败 |
| **游客**（只读）| 偶发 | 查看测试结果但不参与操作 |

**设备：** 桌面浏览器（Chrome / Edge / Safari / Firefox）
**主优化分辨率：** 1440×900（最低支持 1366×768）

### 关键设计挑战

**挑战 1：用例列表的「主战场」体验** ⭐
单项目可达 1 万条用例，需在性能、定位、批量操作三方面均不妥协。
解法：虚拟滚动 + 多级筛选 + 全字段搜索 + 批量操作 + 树形导航联动 + 单页可选 100 条。

**挑战 2：失败详情的「人话化」** ⭐
传统测试平台只展示原始堆栈，开发 OK 但测试人员痛苦。
解法：三层递进——失败摘要（一句话）→ 步骤清单（标出失败步）→ 证据（截图 / API 详情）→ 原始堆栈（折叠）。

**挑战 3：自动→手动接力的状态衔接**
自动跑完后用户容易迷茫"现在该干嘛"。
解法：仅在有待录入用例时显示"手动录入"按钮 + 顶部状态横幅明确指引 + 录入界面支持筛选。

**挑战 4：5 角色的可见性差异**
不同角色看到的页面内容差异大，避免"灰色按钮"困惑。
解法：不可用功能直接不显示（非置灰），顶部明显展示当前角色身份。

**挑战 5：多人协作的并发安全**
Lead 分发用例后多人同时录入，需避免数据覆盖。
解法：显式分配模式——每条用例明确"处理人"字段，他人只读。

### 设计机会

- **机会 1：把测试管理做出"看板感"**——跳出 Excel 风格，借鉴 GitHub Actions / Linear 的卡片视觉，让测试结果一眼看出绿/黄/红
- **机会 2：测试 Lead 的「分发协作」原生支持**——传统平台忽视的协作场景，对应旅程 J4 的混合执行流程
- **机会 3：导入文件预览**——TEA JSON 导入前先预览将要导入什么，降低误操作焦虑
- **机会 4：保存策略「润物细无声」**——手动录入采用"每条草稿即时保存 + 最终确认完成"模式

### 关键决策记录

| 决策项 | 决策 |
|--------|------|
| 手动录入保存策略 | 方案 C —— 每条自动保存草稿 + 最终「确认完成」更新计划状态 |
| Lead 分发协作 | 方案 A —— 显式分配（每条用例指定处理人字段）|
| 分页 | 默认 20 条，可选 20 / 50 / 100 |
| 主题 | 一期支持 2-3 个肤色切换（明亮 / 护眼 / 暗黑）|
| 视觉风格 | 以 Element Plus 为底，参考 Linear 的留白和层次感，配色偏中性蓝灰 |
| 界面语言 | 全中文为主，技术术语保留英文（Flaky、JUnit XML、pytest 等）|

## Core User Experience

### 定义体验

**核心动作：查看报告 → 定位失败**

测试管理平台最高频、最关键的交互是"执行完测试后查看报告并定位失败原因"。测试人员每天多次进入报告页，他们的心理模型是：

> "跑完了吗？过了吗？哪个场景挂了？为什么挂？"

这四个问题必须在 **4 次点击以内** 得到完整回答。报告页的"一目了然 → 精准定位"链路是平台体验的第一优先级。

**与 Apifox 的关键差异——场景级统计 vs 接口级统计：**

Apifox 作为 API 测试工具，以"单个 HTTP 请求"作为统计单元（如 94 个请求通过率 100%）。但本平台以**"测试场景"**为业务单元——一个场景可能包含数十个 HTTP 请求，用户关心的是"3 个场景是否全部通过"，而非"94 个请求是否全部通过"。

> **场景粒度锚定：** 场景以 TEA 输出的 `case_id` 为唯一标识，不以文件为单位。一个 `.py` 文件若包含多个 `case_id`，算多个场景。

### 报告信息架构——四层渐进式设计

基于讨论共识，从 Apifox 的三层升级为**四层**，新增模块分组层以应对几百个场景的导航需求：

**第一层：总览仪表盘（全局统计）**

| 元素 | 说明 |
|------|------|
| 环形图 | 场景通过率，中心显示"通过数/总数"，按状态分色 |
| 指标卡片 | 总场景数、通过数、失败数、错误数、Flaky 数、跳过数 |
| 补充指标 | 总耗时、平均场景耗时、总 HTTP 请求数、总断言数 |
| 执行信息 | 执行时间、执行环境、执行人 |
| 预留入口 | "查看历史趋势"按钮位（MVP 可灰置，为 Test Lead 趋势对比需求预留） |

**状态枚举与颜色体系：**

| 状态 | 含义 | 颜色 | 计入通过率分母？ |
|------|------|------|----------------|
| **passed** | 所有步骤通过 | 绿色 | 是（分子+分母） |
| **failed** | 至少一个步骤断言失败 | 红色 | 是（仅分母） |
| **error** | 非断言异常（fixture 崩溃、环境问题、超时） | 橙色 | 是（仅分母） |
| **flaky** | 首次失败但重试后通过 | 黄色 | 是（仅分母） |
| **skipped** | 被标记跳过（Flaky 标记 / 无脚本 / 条件跳过） | 灰色 | 否 |
| **xfail** | 已知缺陷，预期失败 | 蓝灰色 | 否 |

**通过率公式：**

```
通过率 = passed / (passed + failed + error + flaky) × 100%
```

> skipped 和 xfail 不计入分母，避免通过率虚高或虚低。

**环形图分段：** 六种状态各占一段弧，绿色（passed）为主色调，红色（failed）和橙色（error）最醒目。

**Tab 筛选栏：**

`全部(N) | 通过(N) | 失败(N) | 错误(N) | Flaky(N) | 跳过(N)`

> 计数基于场景数。点击 Tab 筛选后，下方列表联动。

**自动/手动执行来源 Toggle：**

报告顶部提供来源筛选：`全部 | 仅自动化 | 仅手动`，因为自动化和手动结果的置信度不同，Lead 需要分别审视。

**第二层：模块分组视图（导航层）**

> 几百个场景需要按模块组织，否则列表过长难以定位。

| 元素 | 说明 |
|------|------|
| 分组方式 | 按"模块 → 子模块"两级分组，与用例管理的树结构一致 |
| 每组信息 | 模块名称、场景总数、通过/失败/错误/Flaky 计数、模块级通过率、总耗时 |
| 折叠/展开 | 默认折叠，点击展开显示该模块下的场景列表 |
| 排序 | 有失败/错误的模块排在最前面（后端排序） |

**第三层：场景步骤列表（定位层）**

| 元素 | 说明 |
|------|------|
| 场景行 | 场景名称 + 状态标签 + step 数量 + 耗时 + 执行来源标签（自动/手动） |
| 自动化场景展开 | 显示该场景内每个步骤：步骤名称、HTTP 方法标签、URL、状态（✅/❌/⚠️）、耗时 |
| 手动场景 | **不可展开**，仅显示场景行信息（状态 + 录入人 + 备注摘要），无步骤详情 |
| 失败高亮 | 失败步骤红色标记 + 一句话错误摘要 |

> **加载策略：** 展开场景时加载该场景所有 steps 的轻量信息（名称、方法、URL、状态、耗时），不含 request/response body。

**第四层：请求/响应详情面板（证据层）**

| 元素 | 说明 |
|------|------|
| 触发方式 | 点击自动化场景中的某个步骤行，右侧滑出详情面板 |
| Request Tab | Headers、Params、Body（JSON 语法高亮） |
| Response Tab | Status Code、Headers、Body（JSON 语法高亮） |
| 断言 Tab | 断言列表（名称、类型、期望值、实际值、通过/失败） |
| E2E 额外 | 失败截图（缩略图 + 点击放大） |
| 堆栈 Tab | 原始错误堆栈（默认折叠） |
| 开发工具 | 一键复制 curl 命令（方便 Dev 本地复现） |
| 手动场景 | 不适用——手动场景无详情面板 |

> **加载策略：** 点击具体 step 时才加载该 step 的完整 request/response + assertions（懒加载）。单场景 step 最多达 80-90 个，一次性加载全部详情数据量过大。

### 手动场景的差异化展示

自动化场景和手动场景在报告中的展示方式完全不同：

| 维度 | 自动化场景 | 手动场景 |
|------|-----------|---------|
| 可展开 | 展开显示步骤列表 | 不可展开 |
| 详情面板 | 请求/响应 JSON | 无 |
| 场景行信息 | 状态 + step 数 + 耗时 | 状态 + 录入人 + 备注摘要 |
| 来源标签 | `自动` 标签 | `手动` 标签 |
| 证据类型 | HTTP 请求/响应、截图、堆栈 | 仅文字备注 |

### 平台策略

- **设备：** 桌面浏览器优先（Chrome / Edge / Safari / Firefox）
- **主优化分辨率：** 1440×900（最低支持 1366×768）
- **交互方式：** 鼠标 + 键盘，不考虑触屏优化
- **离线能力：** 不需要；但 HTML 导出报告需自包含，可离线浏览
- **响应式：** 不需要移动端适配，但需确保 1366 宽度下布局不破碎

### 零思考交互设计

| 用户意图 | 零思考实现 |
|---------|-----------|
| "跑完了吗？" | 报告页顶部状态横幅：执行中（进度条）/ 已完成（绿色对勾）/ 有失败（红色警告） |
| "整体过了吗？" | 环形图一眼看出——全绿 = 全过，有红/橙/黄 = 有问题 |
| "哪个模块有问题？" | 失败/错误模块自动排最前，红色计数醒目 |
| "哪个场景挂了？" | 展开模块 → Tab 切到"失败" → 直接看到失败场景列表 |
| "为什么挂？" | 展开场景 → 失败步骤红色高亮 → 点击查看请求/响应详情 |
| "给开发看下" | 详情面板一键复制 curl，或导出 HTML 报告发链接 |
| "这次比上次好了吗？" | 历史趋势入口（MVP 预留位置） |

### 关键成功时刻

| 时刻 | 用户感受 |
|------|---------|
| **首次打开报告** | "一眼就知道整体情况"——环形图 + 指标卡片 + 模块分组，不需要滚动 |
| **定位到失败原因** | "4 次点击就找到了"——失败模块 → 展开 → 失败场景 → 步骤详情 |
| **分享给开发** | "不用截图解释了"——一键复制 curl 或导出 HTML |
| **混合执行结束** | "清楚知道哪些是自动跑的、哪些是手动测的"——来源标签 + toggle 筛选 |
| **发现 Flaky** | "黄色标记让我不会忽视不稳定测试"——Flaky 独立计数和颜色 |

### 体验原则

| 原则 | 含义 | 设计指导 |
|------|------|---------|
| **场景优先** | 统计、筛选、导航都以测试场景为单位，HTTP 请求是场景的内部细节 | 环形图统计场景数；通过率基于场景计算 |
| **四层递进** | 总览 → 模块 → 场景 → 详情，每层点击带来明确的信息增量 | 默认只展示总览，用户主动下钻才展开更多 |
| **失败即焦点** | 失败信息在每一层都优先展示、高亮标记、提供快捷入口 | 失败模块/场景排前面；失败步骤红色高亮 |
| **证据即答案** | 请求/响应 JSON、截图、堆栈、curl——让证据自己说话 | JSON 语法高亮；截图可放大；curl 一键复制 |
| **来源透明** | 自动 vs 手动结果置信度不同，展示方式不同，统计可分别查看 | 来源标签 + toggle 筛选；手动场景不可展开 |

## Desired Emotional Response

### 主要情感目标

**核心情感：安心感——"平台可靠，数据准确，我可以放心依赖它做决策"**

测试管理平台不是让人"兴奋"的产品，它是让人"放心"的工具。用户每天打开它，不是为了获得惊喜，而是为了确认一切正常、或者快速知道哪里出了问题。安心感的本质是**可预测性**——用户能准确预期每次操作的结果，每个数字都值得信赖。

这意味着：
- 界面稳定、一致，不搞意外的动效或布局跳动
- 数据加载有明确的过程反馈，不让用户怀疑"是不是卡了"
- 操作结果即时反馈，"我做了什么"和"系统做了什么"始终清晰

### 情感旅程映射

| 阶段 | 场景 | 目标情感 | 设计含义 |
|------|------|---------|---------|
| **每日回访** | "赶紧确认下昨晚自动化跑完了没" | **紧迫 + 目标驱动** | 报告页首屏必须立即呈现最新执行状态，零操作即可回答"跑完了吗？过了吗？" |
| **全部通过** | 环形图全绿，指标正常 | **淡淡的满足** | 不需要庆祝动效。安静的全绿环形图 + 指标卡片就是最好的"一切正常"信号 |
| **部分失败** | 有红色/橙色出现 | **紧迫但不恐慌** | 红色/橙色引导注意力，但不是"报警式"闪烁。清晰的失败计数 + 明确的下钻路径 |
| **大面积失败** | 超过 50% 场景失败 | **冷静的紧迫** | 数字说话，不渲染恐慌。提供"失败模块排序"让用户有条理地逐个排查 |
| **定位到失败原因** | 展开步骤、查看详情 | **确定感** | 每一层下钻都提供明确的信息增量，到达详情时感觉"我找到原因了" |
| **导出/分享报告** | 给团队或领导发报告 | **专业感 + 信任** | 导出的 HTML 报告视觉精良、数据完整，代表用户的专业水准 |
| **系统出错** | 加载失败、操作异常 | **被照顾感** | 错误信息说人话、给出下一步建议，不甩技术堆栈给用户 |

### 微情绪设计

**需要强化的微情绪：**

| 微情绪 | 触发场景 | 设计手段 |
|--------|---------|---------|
| **信任** | 每次看到数据都准确一致 | 数据来源后端计算，不让前端做统计逻辑；通过率精确到小数点后一位；时间戳精确到秒 |
| **确定** | 操作后立即看到结果 | 所有操作（展开、筛选、切换 Tab）200ms 内响应；状态变更有明确的视觉反馈 |
| **掌控** | 面对大量数据不迷路 | 模块分组 + Tab 筛选 + 失败优先排序，用户始终知道"我在哪、我能做什么" |
| **效率** | 重复操作流畅无阻 | 记住用户上次的筛选状态；键盘快捷键支持（如 ↑↓ 切换场景） |

**需要消除的微情绪：**

| 微情绪 | 触发场景 | 消除手段 |
|--------|---------|---------|
| **烦躁（最高优先级）** | 系统响应慢 | Skeleton 骨架屏加载；分层懒加载确保首屏 < 1s 可交互；进度反馈精确 |
| **困惑** | 不知道数字怎么来的 | 通过率公式在 tooltip 中说明；状态含义在图例中解释；鼠标悬停显示具体数字 |
| **压迫** | 几百个场景密密麻麻 | 模块分组默认折叠；合理分页；适当的行间距和留白 |
| **焦虑** | 找不到关注的信息 | 搜索框始终可见；Tab 筛选计数前置；失败/错误自动排最前 |

### 设计启示

**情感 → 视觉语言映射：**

| 情感目标 | 配色 | 动效 | 文案 |
|---------|------|------|------|
| 安心感 | 中性蓝灰为主色，低饱和度，不刺眼 | 无装饰性动效，仅功能性过渡（展开/折叠 200ms ease） | 陈述式，不用感叹号 |
| 淡淡的满足 | 全绿时环形图用柔和的绿色（非荧光绿） | 无庆祝动效，图表静态渲染 | "全部通过"而非"太棒了！全部通过！" |
| 紧迫不恐慌 | 失败用暖红色（#E53E3E），非警报红 | 无闪烁，无弹窗打断 | "8 个场景失败"而非"警告！大量失败！" |
| 专业信任 | 对齐工整、数据密度适中 | 表格排版整齐，数字右对齐 | 技术术语保持准确，不过度简化 |

**加载体验——消除烦躁的核心策略：**

| 场景 | 策略 | 目标 |
|------|------|------|
| 报告首屏 | Skeleton 骨架屏：先渲染布局框架，再填充数据 | 感知加载时间 < 500ms |
| 展开场景 | 展开动画先执行，数据同时加载，loaded 后渐显 | 操作反馈 < 100ms |
| 详情面板 | 面板先滑出（空态），数据加载完后渐入 | 交互响应 < 100ms |
| HTML 导出 | 立即弹出进度条（"生成中 30%..."），完成后自动下载 | 不让用户等待无反馈 |

### 情感设计原则

| 原则 | 含义 |
|------|------|
| **安静即可靠** | 不用动效炫技，稳定一致的界面就是最大的安心感来源 |
| **数据即情绪** | 全绿的环形图自然带来满足，大面积红色自然带来紧迫——让数据自己传递情绪，不需要额外的情绪化 UI 元素 |
| **速度即尊重** | 每一次等待都是在消耗用户的信任。首屏 < 1s 可交互、操作 < 200ms 响应，是底线而非目标 |
| **克制即专业** | 没有 confetti、没有表情包、没有感叹号。测试管理是严肃的工作工具，克制的设计语言传递专业感 |

## 核心用户体验定义（深化）

### 定义体验

**一句话：执行测试，4 次点击找到失败原因。**

这是测试管理平台的"灵魂交互"。用户描述这个平台时会说：

> "跑完测试，打开报告，点几下就知道哪里挂了、为什么挂了。"

### 用户心智模型

测试人员每天面对的核心焦虑是"昨晚的自动化跑完了吗？有没有挂？"他们不是来"管理"的，是来**确认**和**排查**的。

| 用户心理 | 对应设计 |
|---------|---------|
| "我只想知道结果，别让我找" | 报告首屏环形图 + 指标卡，零操作即可回答 |
| "挂了的话，告诉我挂在哪" | 失败模块自动排前 + 失败 Tab 一键筛选 |
| "我要看到证据，不要推测" | 展开到 request/response JSON，证据自己说话 |
| "给开发看的时候别让我截图" | 一键复制 curl + HTML 导出 |

**当前痛点 vs 本平台解法：**

| 痛点 | 现有做法 | 本平台解法 |
|------|---------|-----------|
| pytest 输出难以定位 | 翻几百行日志找 FAILED | 环形图 → 模块 → 场景 → 步骤，逐层聚焦 |
| 报告散落在文件系统 | 去服务器找 HTML 文件 | 平台集中管理，随时打开 |
| 看不到请求详情 | 手动加 print 或开抓包 | 步骤详情面板直接展示 request/response |
| 无法快速分享 | 截图发群 | 一键复制 curl / 导出 HTML / 分享链接 |

### 成功标准

| 标准 | 量化指标 | 原型验证 |
|------|---------|---------|
| 4 次点击到根因 | 从打开报告到看到失败步骤的 request/response ≤ 4 次点击 | ✅ 3-4 次 |
| 首屏零操作 | 打开报告页不需要任何操作即可回答"过了吗" | ✅ 环形图+指标卡首屏可见 |
| 失败定位 < 10 秒 | 从打开报告到锁定失败场景 ≤ 10 秒 | ✅ 失败 Tab + 排序前置 |
| 证据自解释 | 开发拿到详情后无需额外沟通即可排查 | ✅ JSON + 状态码 + curl |

### 交互模式分析

**采用的成熟模式：**

| 模式 | 来源 | 本平台应用 |
|------|------|-----------|
| 环形图总览 | Apifox / Grafana | 报告首屏场景通过率 |
| 折叠式列表 | Apifox / 文件管理器 | 模块 → 场景 → 步骤逐级展开 |
| 右侧滑出面板 | Apifox / IDE | 步骤详情面板，不离开当前页 |
| 内联标签编辑 | Apifox / Notion | 用例详情页属性标签点击下拉修改 |
| 即时编辑 | Notion / Linear | 用例详情页进入即可编辑 |

**本平台的独特组合：**

| 创新点 | 说明 |
|--------|------|
| 场景级统计 | 区别于 Apifox 的接口级统计 |
| 四层递进 | 比 Apifox 多一层模块分组 |
| 自动/手动差异化 | 自动场景可展开查详情，手动场景仅显示备注 |
| 标题栏内联属性 | 用例详情页所有元数据在标题下方标签展示，点击即改 |

### 体验机制（原型验证版）

**核心链路：4 次点击从全局到根因**

```
1. 触发执行
   用例详情页 → 选择环境 → 点击「执行」
   或 测试计划 → 批量执行

2. 结果总览（打开报告页）
   ┌─────────────┬────────────────────────────┐
   │  环形图      │  通过  失败  错误  Flaky   │
   │  87.96%     │  总场景  总耗时  断言数     │
   └─────────────┴────────────────────────────┘
   Tab: 全部(120) | 通过(95) | 失败(8) | ...
   来源: 全部 | 仅自动化 | 仅手动

3. 聚焦模块（点击 1）
   ▶ 审批模块  45场景  5失败  2错误
   ▶ 认证模块  40场景  2失败
   → 失败模块自动排最前

4. 展开场景（点击 2）
   ▼ 审批模块
     ❌ 失败  发布审批-场景3  自动  21steps  4.5s
     ❌ 失败  下线审批-场景7  自动  14steps  3.2s
     ✅ 通过  发布审批-场景1  自动  21steps  2.8s

5. 展开步骤（点击 3）
   ▼ 发布审批-场景3
     ✅ POST /api/approval/list     200  85ms
     ✅ POST /api/approval/submit   200  120ms
     ❌ POST /api/approval/review   500  230ms  ← 红色高亮
     ✅ GET  /api/approval/status    200  60ms

6. 查看证据（点击 4）
   → 右侧 Drawer 滑出
   Request / Response / 断言 Tab
   + [复制 curl 命令]
```

### 脚本管理方式

**决策：方案 A — Git 拉取 + git worktree 并发隔离**

经 Winston（架构）、Amelia（开发）、Murat（测试）三方讨论一致确认。

**架构设计：**

```
存储层：/workspace/repos/{project_id}/     ← git clone/pull 主目录（单一事实来源）
执行层：/workspace/runs/{execution_id}/    ← git worktree 创建的隔离快照
清理层：执行完成后异步清理 runs 目录
```

**关键设计要点：**

| 要点 | 做法 |
|------|------|
| 多项目隔离 | 每个项目独立 Git 仓库目录 |
| 并发安全 | `git worktree add --detach` 零拷贝创建隔离执行目录 |
| 脚本更新 | 点击"更新脚本" → `git pull`，记录 commit SHA |
| 执行追溯 | 每次执行记录 commit SHA，任意时间可精确复现 |
| 环境变量 | `subprocess.run(env=platform_vars)` 注入，覆盖脚本默认值 |
| Windows | `pathlib.Path` 全程 + `core.longpaths=true` |
| 故障恢复 | 删目录重新 clone |

**否决方案 B（数据库存储）的理由：**
- pytest 依赖文件系统目录结构（conftest.py 层级、fixture 依赖链），从数据库重建目录风险极高
- 本质上是重造一个劣质版本控制系统，代码量是方案 A 的 3 倍
- fixture 依赖链初期缺陷率是方案 A 的 5-8 倍

**方案 C（混合：Git + 单文件覆盖）作为 v2 预留，一期不做。**
