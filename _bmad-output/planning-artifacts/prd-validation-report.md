# PRD 校验报告

**校验对象：** `_bmad-output/planning-artifacts/prd.md` (v2.1)
**校验日期：** 2026-04-13
**总体评分：** B-（核心需求清晰，但缺少成功标准、用户旅程、NFR 三个关键支柱）

---

## 校验结论摘要

PRD 在**功能描述层面表达清晰、信息密度高**，但在**可追溯性、可度量性、用户视角**三个维度有明显缺失。这些缺失会在后续的 UX 设计、架构设计和 Story 拆分阶段造成困扰。

**建议：** 修复 4 个 Critical 问题后再进入下一阶段。

---

## 一、Critical 问题（必须修复）

### C1：缺少「成功标准」章节（Success Criteria）

**问题：** 1.2 节只写了产品目标，没有可度量的成功指标。

**影响：** 后续无法判断"产品做对了没有"，UX 和架构无法对齐成功标准。

**建议补充：**
```markdown
## 成功标准

### 业务指标
- 平台上线 3 个月内承接至少 3 个项目的测试管理
- 平均每月手动用例录入数量 ≥ 100 条
- 自动化用例从 TEA 导入到平台可执行的全流程 ≤ 10 分钟

### 体验指标
- 用例列表加载（1000 条）< 2 秒
- 测试计划执行结果回写延迟 < 5 秒
- 报告 HTML 导出（500 条用例）< 30 秒
```

---

### C2：完全缺失「用户旅程」章节（User Journeys）

**问题：** PRD 没有任何"作为 X 角色，我想 Y，以达成 Z"的旅程描述。

**影响：**
- UX 设计阶段无法理解用户操作流
- 难以判断功能间的交互连贯性
- Story 拆分时缺少端到端视角

**建议补充至少 5 条核心旅程：**
1. **测试人员**：从 TEA 输出导入用例 → 创建自动化计划 → 执行 → 查看报告
2. **测试人员**：手动创建用例 → 创建手动计划 → 逐条录入结果
3. **项目管理员**：创建项目 → 配置仓库 → 添加成员 → 更新脚本
4. **测试 Lead**：创建混合计划 → 自动跑完后接力手动 → 导出报告
5. **系统管理员**：创建用户 → 分配角色 → 绑定项目

---

### C3：完全缺失「非功能需求」章节（NFR）

**问题：** PRD 没有性能、并发、安全、兼容性、数据保留等任何 NFR 描述。

**影响：** 架构师无法做关键技术决策（数据库选型已有但缺少指导参数、缓存策略、横向扩展、备份等）。

**建议补充：**
```markdown
## 非功能需求

### 性能
- 用例列表加载 1000 条 < 2 秒（P95）
- 测试计划执行调度延迟 < 1 秒
- 报告 HTML 导出 500 条用例 < 30 秒

### 并发与容量
- 支持单平台同时承载 50 个项目
- 单项目支持 10000 条用例
- 同时执行 5 个自动化计划（不同项目）

### 安全
- 密码使用 bcrypt 加密存储
- Session 超时 8 小时
- 操作日志保留至少 1 年
- 钉钉 Webhook URL 加密存储

### 兼容性
- 浏览器：Chrome 100+ / Edge 100+ / Firefox 100+
- 部署：Linux（Ubuntu 22.04+ / CentOS 8+）/ Windows Server 2019+

### 数据保留
- 执行历史永久保留（含已归档用例）
- 操作日志保留 3 年
- 报告文件保留 90 天后自动清理（可下载至本地）
```

---

### C4：功能需求缺少 ID 和验收标准

**问题：** 3.x 节描述了功能行为，但没有：
- FR-XXX 编号（导致无法在 Story / 测试中追溯）
- 每条需求的明确验收标准（Given/When/Then 或类似）

**影响：**
- Story 拆分时无法精确引用"实现 FR-012"
- QA 无法编写需求覆盖矩阵
- 修改需求时无法确认变更范围

**建议方式：**
```markdown
### FR-CASE-001：用例导入

**描述：** 用户上传 tea-cases.json 文件，平台批量创建/更新用例。

**验收标准：**
- AC1：上传非 JSON 文件时返回错误"文件格式不支持"
- AC2：JSON 中缺少必填字段（id/type/title）的条目跳过并报告
- AC3：导入完成后展示 4 个数字：新增 / 更新 / 脚本已移除 / 跳过
- AC4：导入过程中页面不可关闭，需进度提示
```

最少需要为每个 3.x 子节配 1-3 个 FR-ID。

## 二、Important 问题（建议修复）

### I1：「执行引擎」章节命令耦合 pytest，未抽象出执行接口

**位置：** 3.7.1

**问题：** 执行命令直接写死 pytest，未来扩展（比如支持 jest、go test）需要改 PRD 而不是仅改实现。

**建议：** 描述层提升到"调用对应语言/框架的测试运行器"，pytest 作为一期的具体实现写入实现说明。

---

### I2：用例字段「子模块」概念出现但未定义

**位置：** 3.3.1 提到"三级树形结构：测试类型 → 模块 → 子模块"，但 3.3.2 用例字段中只有「模块」，无「子模块」字段。

**影响：** 实现时不清楚子模块如何归属（一个用例属于子模块？子模块属于模块？数据模型怎么设计？）

**建议：** 明确以下任一：
- 取消「子模块」概念，仅两级（类型 → 模块）
- 增加「子模块」字段，并说明子模块管理方式

---

### I3：「告警级别」逻辑不闭环

**位置：** 3.5

**问题：** 定义了 P0/P1/P2/P3 告警级别基于失败率，但没说：
- 多次同样级别告警是否合并？
- 告警钉钉消息卡片设计是什么？
- 告警是否有去重时间窗口？

**建议补充。**

---

### I4：「计划状态」未覆盖手动计划的暂停/中止

**位置：** 3.6.3

**问题：** 状态流转图的"已暂停"标注"熔断触发"，但手动计划没有熔断概念。手动计划是否能暂停？测试人员中途退出怎么处理？

**建议：** 拆分为「自动化计划状态机」和「手动计划状态机」分别说明，或明确手动计划无暂停态。

---

### I5：「环境配置」与项目隔离的关系不明

**位置：** 3.4

**问题：** 环境是全局配置，但用户与权限是项目级隔离。问题：
- 项目 A 的开发能看到为项目 B 创建的环境吗？
- 不同项目的"测试环境"如何区分？（同名冲突）

**建议：** 明确环境是"全局可见、全局编辑"，或"全局命名空间但绑定项目"。
