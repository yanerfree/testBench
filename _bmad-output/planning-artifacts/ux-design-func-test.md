## Executive Summary

### Project Vision

功能场景测试模块的 UX 使命：让"需求文档 → 可执行的功能测试场景用例"这条 AI 链路**全程可见、可控、可信**。用户不是在"等 AI 出结果"，而是在一条五阶段向导流上与 AI 协作——每一步看得到 AI 在干什么、改得了 AI 的中间产物、查得到每条结果的来龙去脉。体验目标一句话：**把两天的活变成半小时，且这半小时里用户从不迷路、从不白等、从不怀疑数据是假的。**

### Target Users

- **测试工程师（主力，不写代码）**：每周多次使用，从需求文档生成用例是核心工作流；对"AI 黑盒"天然警惕，需要看见中间产物才敢信
- **测试 Lead（评审者）**：批量评审是高频动作，效率敏感（键盘流、排序、批量）；需要覆盖矩阵和统计视图回答"测全了吗/AI 好用吗"
- **开发工程师（Claude Code 通道）**：不进平台 UI 生成，但其产出会出现在平台的评审队列——双通道产物在 UI 上必须无差别
- **管理者（只读）**：报告与覆盖视角，几分钟内看懂质量结论

### Key Design Challenges

1. **长时序 AI 任务的信任感**：需求提取 <60s、批量展开分钟级——刷新/断线/隔天回来都不能丢进度、丢产出（FR61/63：任务持久化 + SSE 回放）
2. **中间产物的可编辑呈现**：场景模型（流程/状态机/角色矩阵/测试点）信息密度高，要可编辑但不能做成图形化编辑器（FR69 明确表格化最小交互）——挑战是"表格承载结构化模型还能让人看出业务逻辑"
3. **评审吞吐量**：一次生成 40+ 条用例，评审不能变成新瓶颈（FR64 键盘流 + 评分排序 + 批量）
4. **与既有页面的一致性**：用例落在既有用例管理（CaseManagement/CaseDetail），新增的生成/评审/矩阵界面必须与平台清新空灵风（全局渐变/色板）和 AntD 习惯无缝衔接

### Design Opportunities

1. **覆盖矩阵作为"指挥台"**：不只是报表——单元格直达补充生成（FR66），让"发现缺口→补齐"变成一次点击，这是竞品没有的闭环体验
2. **生成档案时间线**：每条用例可解释（哪个模型/哪版 prompt/为何被拒过），把"AI 可信度"做成看得见的 UI 资产（FR57/65）
3. **双栏生成页**：左侧进度流 + 右侧产出面板实时长出用例卡片（FR62）——把等待变成"看着结果长出来"的正反馈
4. **诚实 UI 品牌**：真实空态、失败原因必现、无假数据（FR68）——对标项目的 mock 数据剧场是反面教材，"数据可信"本身就是体验差异化

## Core User Experience

### Defining Experience

**核心动作：把一份需求文档变成一批已审核的场景用例。** 这条主流程（输入 → 需求点确认 → 场景模型确认 → 生成 → 评审）是模块的心脏，五个阶段在**一个页面**内推进——用户永远知道自己在第几步、下一步是什么、随时可以离开再回来。

### Platform Strategy

- Web 桌面端（1280px+），鼠标 + 键盘；评审工作台为键盘优先设计
- 复用平台 SPA 框架（React 19 + AntD + 清新空灵风全局样式），无移动端/离线需求
- 长任务依赖 SSE；断线重连与刷新回放是平台级交互契约（FR63）

### Effortless Interactions

- **零配置起步**：进入生成页粘贴文档即可开始，模块/API/业务规则均为可选增强项
- **确认即前进**：需求点和场景模型页面的主按钮永远是"确认，下一步"，改动自动保存（blur 保存，与接口测试模块一致）
- **跳过确认**：熟练用户一键"跳过确认直接生成"（FR69），模型仍生成并折叠可查
- **评审两击完成**：a=通过；r=拒绝→理由分类弹层（枚举一击+可选文本）→回车提交
- **缺口一键补**：矩阵单元格 hover 出现"+补充生成"，点击即发起，不离开矩阵页

### Critical Success Moments

1. **首次生成的前 60 秒**：需求点带原文引用逐条流出——用户第一次确认"AI 真读懂了我的文档"（信任建立时刻）
2. **场景模型确认页**：用户发现 AI 漏了一条状态流转、手工补上——"我在掌控"的时刻（此交互若卡顿或丢改动，信任崩塌）
3. **批量生成时的双栏页**：右栏用例卡片逐条长出、带评分徽标——等待变成收获感
4. **评审 40 条用例只花 10 分钟**：键盘流 + 评分排序生效的时刻
5. **覆盖矩阵全绿**：用户截图发群里汇报"这个需求测全了"——传播时刻

### Experience Principles

1. **过程可见（Show the work）**：AI 的每个中间产物都渲染成可读的 UI，绝不"转圈→结果"
2. **永不丢活（Never lose work）**：任务/中间产物/改动全部持久化，刷新、断线、隔天回来都在原地
3. **键盘是快车道，鼠标是完整道**：所有操作鼠标可达，高频操作键盘更快
4. **诚实呈现（Honest UI）**：失败给原因、空态是真空、数字是真数、不承诺后端没有的动作
5. **一致优先于新颖**：能复用平台既有交互（目录树/表格/Tag/抽屉/弹窗）就不发明新模式

## Desired Emotional Response

| 阶段 | 目标情绪 | 反面（要避免的） | 设计手段 |
|------|---------|----------------|---------|
| 输入需求 | 轻松起步："扔进来就行" | 被表单吓退 | 单个大输入区，增强项折叠 |
| 需求点/模型确认 | 掌控感："AI 是我的下属" | 被动接受黑盒 | 逐条原文引用、行级编辑、我改的字段有视觉标记 |
| 批量生成 | 收获感："结果在长出来" | 干等焦虑 | 双栏实时卡片流 + 进度数字 + 可离开提示 |
| 生成中断 | 安心："什么都没丢" | 恐慌重来 | 已完成部分绿色保留 + 明确断点 + 一键续跑 |
| 评审 | 心流："唰唰唰过" | 繁琐负担 | 键盘流 + 评分排序 + 批量 + 进度计数 |
| 看矩阵/报告 | 底气："我能证明测全了" | 心虚拍脑袋 | 矩阵全景 + 零覆盖醒目告警 + 可追溯链接 |

**情绪主基调**：专业工具的"沉稳可靠"（配平台清新空灵风的轻盈视觉），不做拟人化 AI 角色、不用娱乐化动效——目标用户是每天用它交付工作的工程师。

## 交互规范

### 审核状态与评分色

| 元素 | 规则 |
|------|------|
| 待审核 | `<Tag color="processing">待审核</Tag>` |
| 已审核 | `<Tag color="success">已审核</Tag>` |
| 已拒绝 | `<Tag color="error">已拒绝</Tag>`（hover 显示理由） |
| 质量评分 | ≥85 绿 / 70-84 蓝 / <70 橙，数字徽标；未评分显示 "—" 非 0 |
| 维度 Tag | 正向绿 / 异常红 / 边界橙 / 权限紫 / 数据青 / 状态蓝（与矩阵列一致） |
| 任务阶段 | ○未开始 ◐进行中 ●完成 ✕失败（同接口测试模块圆圈语义） |

### 键盘映射（评审工作台）

j/k 上下 · a 通过 · r 拒绝 · e 编辑 · space 勾选 · shift+a 批量通过已勾选 · 🔍/o 查看原文引用 · ? 快捷键帮助浮层；输入框聚焦时快捷键自动失效。

### SSE 与任务恢复契约（全模块统一）

1. 页面进入/重连：先 GET 任务当前状态全量渲染，再订阅 SSE 增量（FR63）
2. SSE 断开 >5s：顶部细条提示"连接已断开，尝试重连…"，不弹 Modal
3. 任务进行中离开页面：无需确认（自动保存），任务中心行显示实时进度
4. 失败项一律带一行人话原因 + 可行动按钮（续跑/重试/查看详情）

### 编辑与保存

- 需求点/测试点/用例字段：行内编辑，blur 自动保存（与接口测试模块一致）
- 用户编辑过的 AI 产物：行首 ✎ 蓝色标记持久显示（支撑"编辑率"度量的可见化）
- 批量展开前的确认按钮：文案永远带数量（"开始生成 42 条用例"），无惊喜原则

### 诚实 UI 检查单（FR68 落地）

- 所有列表空态配一句"下一步该做什么"+ 主按钮，禁止占位假数据
- 失败态必含：失败环节 + 人话原因 + 重试/续跑动作
- 统计数字必须来自真实查询；未实现的功能不出现按钮（宁缺勿灰）
