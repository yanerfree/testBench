# 产品需求文档（PRD）
# 测试管理平台

**版本：** 3.5
**日期：** 2026-04-15
**状态：** 草稿

---

## 一、产品概述

### 1.1 背景

团队使用 TEA（BMAD Test Architect）生成自动化测试脚本，但缺乏统一的平台来管理用例、执行测试计划和查看结果。当前问题：
- 用例散落在脚本文件中，无法结构化查看
- 手动测试用例缺少统一管理入口
- 测试执行结果无法集中汇总和追踪

### 1.2 目标

构建一个轻量级测试管理平台，以 TEA 生成的用例文件为主要数据来源，同时支持手动录入，提供用例管理、测试计划执行（自动+手动）、结果查看与报告导出能力。

### 1.3 范围

**一期（核心功能）：**
- 用户与权限（5 级角色）
- 项目管理、模块管理
- 用例导入与手动维护
- 分支配置（多版本/多分支用例管理）
- 变量与环境配置（全局变量 + 环境变量）、通知渠道配置
- 测试计划创建与执行（自动化 + 手动）
- 执行结果与报告（下钻 + 导出）
- 操作日志

**二期：**
- Cron 定时调度
- 执行 Diff 对比、质量趋势曲线

**暂不实现：**
- CI 集成（GitLab Webhook）
- Git 提交关联矩阵
- Flaky 自动检测与统计

## 二、成功标准

### 2.1 业务指标

| 指标 | 目标值 | 测量方式 |
|------|-------|---------|

### 2.2 体验指标

| 指标 | 目标值 |
|------|-------|
| 用例列表加载（1000 条） | < 2 秒（P95） |

| 报告 HTML 导出（500 条用例） | < 30 秒 |
| 报告 Excel 导出（500 条用例） | < 15 秒 |

### 2.3 技术指标

| 指标 | 目标值 |
|------|-------|

## 三、用户与权限

### 3.1 角色定义（5 级）

| 角色 | 可见范围 | 项目配置 | 用例/执行/结果录入 | 环境/渠道配置 | 用户管理 |
|------|---------|---------|-------------------|-------------|---------|
| **系统管理员** | 所有项目 | ✅ | ✅ | ✅ | ✅ |
| **项目管理员** | 绑定的项目 | ✅（所属项目） | ✅ | ✅ | ❌ |
| **开发** | 绑定的项目 | ❌ | ✅ | ✅ | ❌ |
| **测试** | 绑定的项目 | ❌ | ✅ | ✅ | ❌ |
| **游客** | 绑定的项目 | ❌ | ❌ 只读 | ❌ 只读 | ❌ |

### 3.2 关键规则

- 系统管理员可看所有项目所有数据，可操作用户管理
- 除系统管理员外，其他角色必须绑定到项目后才能看到 / 操作该项目数据
- 项目管理员仅能管理绑定项目的配置和成员
- 开发 / 测试 可操作用例、执行、结果、录入，但不可改项目配置
- 游客仅查看

### 3.3 登录

- 方式：用户名 + 密码
- 系统管理员在后台创建用户并分配角色和项目绑定

## 四、用户旅程

### 旅程 J1：测试人员 — 从 TEA 导入到执行报告（核心旅程）

**角色：** 测试 / 开发
**目标：** 把 TEA 生成的用例集快速跑起来并查看结果

```
1. 测试人员从 TEA 输出获得 tea-cases.json
2. 登录平台 → 进入项目 → 进入「测试用例」页 → 选择分支 → 点击「导入」
3. 上传 JSON → 平台解析 → 展示「新增 X / 更新 Y / 脚本已移除 Z」
4. 切换到「测试计划」页 → 创建「自动化计划」
   - 选择 E2E 类型 → 勾选刚导入的用例 → 选择目标环境 → 选择钉钉渠道
5. 点击「执行」→ 平台逐条调用脚本 → 实时显示执行进度
6. 执行完成 → 进入计划详情页查看结果
7. 失败用例展开看 pytest 错误堆栈
8. 点击「导出 HTML」→ 下载报告分享给团队
```

**关键交互点：** 导入摘要清晰、执行进度实时、失败定位直观。

---

### 旅程 J2：测试人员 — 手动用例与手动计划

**角色：** 测试
**目标：** 录入和执行没有自动化脚本的手动测试

```
1. 测试人员新建用例 → 填写标题、模块、优先级、前置条件、步骤、预期
2. 不填 script_ref → 自动化状态显示「待自动化」
3. 创建「手动计划」 → 选 API 类型 → 勾选这批手动用例
4. 点击「执行」→ 进入手动录入模式
5. 逐条用例：阅读步骤 → 实际操作 → 录入结果（通过/失败）+ 备注
6. 全部录入完成 → 计划状态变为「已完成」
7. 导出 Excel 报告
```

**关键交互点：** 步骤展示清晰、录入操作单手键盘可完成。

---

### 旅程 J3：项目管理员 — 项目创建与脚本同步

**角色：** 项目管理员
**目标：** 把一个新项目接入平台

```
1. 项目管理员登录 → 进入「项目管理」
2. 创建项目 → 填写：项目名称、Git 地址、脚本路径、成员列表
3. 项目创建成功 → 系统自动创建默认分支配置（分支 `main`）
4. 如需管理多版本 → 创建新分支配置 → 填写分支配置名称和 Git 分支
5. 进入「测试用例」页 → 选择目标分支 → 点击「更新脚本」→ 平台从 Git 拉取该分支最新代码
6. 平台展示更新摘要：新增 N 个文件 / 修改 M / 删除 K
7. 自动检测 script_ref 失效用例 → 标记「脚本已移除」
8. 分支就绪，团队成员在用例管理页选择该分支开始工作
```

**关键交互点：** Git 拉取过程透明、失败有明确错误提示。

---

### 旅程 J4：测试 Lead — 混合计划闭环

**角色：** 测试（兼任 Lead）
**目标：** 一次性完成自动+手动混合执行

```
1. 测试 Lead 创建「自动化计划」→ 勾选包含自动化和待自动化的混合用例
2. 点击「执行」→ 平台先跑「已自动化」用例（约 20 分钟）
3. 自动执行结束 → 计划进入「待手动录入」阶段 → 钉钉通知 Lead
4. Lead 进入计划详情 → 看到自动结果 + 待手动用例列表
5. 分配「待自动化」用例给 2 个测试人员手动执行
6. 测试人员逐条录入结果
7. Lead 等所有用例完成 → 导出统一 HTML 报告（含自动+手动）
```

**关键交互点：** 自动→手动衔接顺畅、Lead 能看到全局进度。

---

### 旅程 J5：系统管理员 — 用户与项目治理

**角色：** 系统管理员
**目标：** 新增团队成员并分配项目

```
1. 系统管理员登录 → 进入「用户管理」
2. 创建新用户 → 填写：用户名、密码、角色（测试/开发/...）
3. 进入「项目管理」 → 选择目标项目 → 绑定该用户
4. 新用户登录 → 自动看到所属项目数据
```

**关键交互点：** 角色分配后立即生效、用户体验无感知。

---

### 旅程 J6：游客 — 查看项目报告

**角色：** 游客（如产品经理、开发负责人）
**目标：** 查看测试结果但不参与操作

```
1. 游客登录 → 看到所属项目列表
2. 进入项目 → 用例 / 计划 / 报告 全部只读
3. 浏览最近一次执行的报告 → 查看通过率、失败明细
4. 下载 HTML 报告备份
```

**关键交互点：** 所有"操作"按钮（编辑/执行/导入）对游客隐藏。

## 五、功能需求

> **说明：** 每条 FR 包含编号、描述和验收标准（AC）。FR-ID 用于在 Story、测试用例中追溯。

### 5.1 项目管理

#### FR-PROJ-001：创建项目

**描述：** 系统管理员或项目管理员创建新项目。

**字段：** 项目名称、Git 仓库地址、脚本基础路径（绝对路径，各分支配置目录自动派生）、项目成员列表（含角色）

**验收标准：**
- AC1：项目名称在系统内唯一，重名提示错误
- AC2：脚本存放路径不存在时，平台自动创建该目录
- AC3：Git 地址格式校验（支持 `git@host:user/repo.git` 和 `https://`）
- AC4：成员列表至少包含创建者本人

> **说明：** 项目不再直接绑定分支，分支由分支配置管理（见 FR-BRANCH）。创建项目时系统自动创建默认分支配置（名称 `default`，分支 `main`），用户无需手动操作即可开始工作。

#### FR-PROJ-002：项目列表与编辑

**描述：** 显示用户所属项目，支持修改成员、仓库地址、脚本路径。

**验收标准：**
- AC1：系统管理员看到全部项目，其他角色仅看绑定项目
- AC2：修改 Git 地址后，下次更新脚本生效
- AC3：移除成员后，该成员立即失去项目访问权
- AC4：项目卡片展示当前活跃分支配置数量、总用例数、最近一次执行状态

#### FR-PROJ-003：更新脚本

**描述：** 在用例管理页面，用户选择分支后，点击「更新脚本」按钮触发，从 Git 仓库拉取该分支配置对应分支的最新代码到脚本存放路径，并更新该分支的用例状态。

**验收标准：**
- AC1：「更新脚本」按钮在用例管理页面的分支上下文中显示，触发后进度条显示 git pull 状态
- AC2：完成后展示摘要：新增 N / 修改 M / 删除 K 文件
- AC3：自动遍历该分支内用例的 script_ref，失效的标记为「脚本已移除」
- AC4：拉取失败（如认证错误）时显示错误信息但不影响平台其他功能
- AC5：不同分支的脚本更新互不影响

### 5.1.1 分支配置

> **设计背景：** 同一产品/项目存在多个版本（如 v1.0、v2.0），每个版本对应不同的 Git 分支，拥有独立的用例集和脚本目录。引入分支配置层级避免为每个版本创建独立项目导致项目膨胀。分支配置仅管理用例和脚本同步，测试计划与报告为项目级资源。

#### FR-BRANCH-001：创建分支配置

**描述：** 在项目下创建分支配置，每个分支配置绑定一个 Git 分支。

**字段：** 分支配置名称、Git 分支名（默认 `main`）、描述（可选）、状态（活跃/已归档）

**验收标准：**
- AC1：分支配置名称在项目内唯一
- AC2：分支配置名称仅允许字母、数字、连字符和下划线（`[a-zA-Z0-9_-]`，最长 50 字符），因为名称用于文件系统路径派生
- AC3：同一项目内多个分支配置可绑定不同分支
- AC4：同一项目内允许多个分支配置绑定同一分支（不同阶段的测试）
- AC5：创建项目时自动创建一个默认分支配置（名称 `default`，分支 `main`）
- AC6：分支配置的 Git 分支支持修改，修改后下次"更新脚本"时拉取新分支代码
- AC7：分支配置的脚本目录自动派生为 `{项目脚本基础路径}/{分支配置名称}/`，无需手动指定
- AC8：分支配置名称创建后不可修改（因为关联文件系统路径）

#### FR-BRANCH-002：分支列表与切换

**描述：** 在用例管理页顶部显示分支选择器，切换分支显示该分支的用例。

**验收标准：**
- AC1：用例管理页顶部展示分支选择器，列出所有分支配置，按状态分组（活跃 / 已归档）
- AC2：每个分支配置选项显示：名称、Git 分支、用例数
- AC3：切换分支后，用例列表刷新为该分支的用例数据
- AC4：已归档分支配置的用例数据只读

#### FR-BRANCH-003：分支归档

**描述：** 分支配置完成后可归档，归档后数据只读但可查看。

**验收标准：**
- AC1：活跃分支配置可手动归档
- AC2：归档后该分支的用例仍可查看但不可修改
- AC3：归档操作可逆（可恢复为活跃）
- AC4：项目至少保留一个活跃分支配置

#### FR-BRANCH-004：分支数据隔离

**描述：** 不同分支配置的用例和目录结构隔离。

**验收标准：**
- AC1：用例归属于分支配置，不同分支配置的用例独立管理
- AC2：目录结构按分支配置隔离（不同分支配置可有不同的模块树）
- AC3：导入 tea-cases.json 时，数据写入当前分支配置

#### FR-BRANCH-005：跨分支用例复制

**描述：** 用户可从源分支配置批量选择用例，复制到目标分支配置。避免新分支从零开始。

**验收标准：**
- AC1：目标分支的用例列表页提供"从其他分支导入"入口
- AC2：用户先选源分支配置，再通过模块/关键词筛选用例，支持全选
- AC3：复制产生新的用例记录（新 UUID、新用例 ID 编号），module 归属不变
- AC4：目标分支配置中不存在的目录结构自动创建
- AC5：复制完成后显示摘要："成功复制 X 条用例"
- AC6：复制是深拷贝（步骤、预期结果等一并复制），不建立引用关系

### 5.2 模块管理

#### FR-MOD-001：模块定义

**描述：** 项目内创建模块，含名称和唯一缩写码。模块用于用例归类。

**验收标准：**
- AC1：模块缩写码全大写，仅含 A-Z 和数字
- AC2：模块缩写码在项目内唯一
- AC3：模块缩写码在任意用例创建或导入后变为只读
- AC4：模块名称可修改，不影响用例 ID

#### FR-MOD-002：子模块定义

**描述：** 模块下可建立子模块，用于更细粒度的归类。建议按"页面 / 子页面 / 操作"或"功能 / 子功能"维度划分，因为每个测试场景独立成文件，模块越细越便于管理和定位。

**字段：**
- 子模块名称（必填，模块内唯一）
- 描述（可选）

**子模块两种来源：**
- **TEA 自动生成**：导入 `tea-cases.json` 时，根据每条用例的 `submodule` 字段自动创建（如不存在）
- **用户手动创建**：在模块管理页手动创建

**验收标准：**
- AC1：子模块归属一个模块
- AC2：子模块仅有名称，无缩写码
- AC3：用例 ID 仍以模块为粒度（`TC-{MODULE}-{seq5}`），不含子模块标识
- AC4：子模块为可选——用例可直接挂在模块下不指定子模块
- AC5：子模块支持删除（下方无用例时）和重命名
- AC6：导入时遇到不存在的子模块自动创建，并在导入摘要中提示"新建子模块 N 个"

#### FR-MOD-003：用例树结构

用例展示采用三级树形结构：

```
测试类型（API / E2E）
  └─ 模块（如 AUTH）
       └─ 子模块（如 LOGIN，可选）
            └─ 用例（TC-AUTH-00001）
```

**验收标准：**
- AC1：左侧 Tab 切换 API / E2E 类型
- AC2：树形可展开/收起
- AC3：未指定子模块的用例直接显示在模块节点下

### 5.3 用例管理

#### FR-CASE-001：用例数据结构

**字段：**

| 字段 | 说明 |
|------|------|
| 用例 ID | 平台主键，格式 `TC-{MODULE}-{seq5}`（如 `TC-AUTH-00001`） |
| 来源 | `导入` / `手动` |
| tea_id | 来源为「导入」时保留 TEA 生成的可读 ID（格式：`{module}_{slug}`），仅用于追溯 |
| 标题 | 用例名称 |
| 类型 | `API` / `E2E` |
| 模块 | 关联模块缩写码 |
| 子模块 | 可选 |
| 优先级 | P0 / P1 / P2 / P3 |
| 前置条件 | 文本列表 |
| 测试步骤 | 有序步骤列表（序号 + 操作描述） |
| 预期结果 | 文本 |
| 自动化状态 | 待自动化 / 已自动化 / 脚本已移除 / 已归档 |
| script_ref | 脚本文件路径 + 函数名（可选） |
| Flaky 标记 | 是 / 否（手动标记） |
| 备注 | 可选 |

**验收标准：**
- AC1：用例 ID 格式 `TC-{MODULE}-{seq5}`，序号在模块内单调递增
- AC2：用例 ID 在创建时由系统建议默认值，**用户可在保存前覆盖**（仍需符合格式且模块内唯一）
- AC3：用例保存后，ID 字段在编辑界面变为只读，不可修改
- AC4：标题非空，最长 200 字符
- AC5：类型仅支持 `API` 或 `E2E`
- AC6：步骤列表至少 1 条
- AC7：来源字段不可手动修改（创建时自动填入）

> **三套标识的关系：**
> - **`TC-{MODULE}-{seq5}`** —— 平台主键，用户可见的用例编号
> - **`tea_id`**（如 `auth_login_redirect_to_dashboard`）—— TEA 端可读 ID，用于反查"TEA 用例对应平台哪条用例"
> - **`script_ref.file`** —— 脚本文件路径，用于执行引擎定位脚本（一场景一文件，路径全局唯一）
>
> 三者互不冲突：**导入匹配用 `tea_id`**（唯一标识，不因脚本重命名/移动而变化），`script_ref.file` 用于执行引擎定位脚本，`TC-` 编号始终是平台主键。

#### FR-CASE-002：用例导入

**描述：** 上传 `tea-cases.json` 批量导入或更新用例。

**验收标准：**

### 5.4 变量与环境配置

> **设计参考：** Apifox 变量体系。分两层管理——全局变量（跨环境通用）和环境变量（按环境区分），环境变量优先级高于全局变量。

#### FR-ENV-001：全局变量管理

**描述：** 全局变量为系统级键值对，不区分环境，所有环境下均生效。适用于公共 token、通用超时配置等不随环境切换而变化的参数。

**验收标准：**
- AC1：全局变量为键值对列表，key 在全局范围内唯一
- AC2：值支持纯文本（一期不区分敏感字段）
- AC3：游客只读，其他角色可创建和编辑
- AC4：无数量上限

#### FR-ENV-002：环境管理

**描述：** 环境用于区分不同的部署目标（如开发、测试、预发布、生产），每个环境拥有独立的环境变量集合。环境是**全局可见、全局编辑**的资源，与项目无绑定关系。

**验收标准：**
- AC1：环境名称在系统内唯一
- AC2：每个环境包含独立的环境变量（键值对列表），无数量上限
- AC3：环境变量值支持纯文本（一期不区分敏感字段）
- AC4：游客只读，其他角色可创建和编辑
- AC5：环境被任何计划引用后不可删除（先解除引用）
- AC6：支持环境复制（快速创建相似环境）

> **说明：**
> - 任何项目创建测试计划时都能选择系统中的所有环境
> - 任何角色（除游客）都可以维护环境列表和全局变量
> - 如需在命名上区分项目用途，可由用户自行约定（如 `proj-a-staging`），平台不强制

#### FR-ENV-003：变量优先级与注入规则

**描述：** 执行测试时，平台将全局变量和选定环境的环境变量合并后注入测试进程。

**优先级（从高到低）：**
1. **环境变量** — 当前选定环境的变量，最高优先级
2. **全局变量** — 跨环境通用变量
3. **脚本内配置** — 测试脚本自身的默认值，最低优先级

**验收标准：**
- AC1：同名 key 时，环境变量覆盖全局变量
- AC2：环境变量中不存在的 key，自动使用全局变量的值
- AC3：合并后的变量以 `os.environ` 方式注入测试进程
- AC4：执行报告记录实际使用的环境名称和变量快照（一期保存全量值供内部排查使用，二期引入敏感字段标记后对外接口自动脱敏）

### 5.5 通知渠道配置

#### FR-NOTIF-001：渠道管理

**描述：** 全局钉钉 Webhook 配置。

**验收标准：**
- AC1：渠道名称在系统内唯一
- AC2：Webhook URL 格式校验
- AC3：Webhook URL 加密存储（对前端展示时遮罩）
- AC4：游客只读，其他角色可创建和编辑

#### FR-NOTIF-002：钉钉通知

**描述：** 在两个特定场景向选定的钉钉渠道推送消息。**不区分告警级别。**

**触发场景：**
1. **自动化计划执行完成**：所有自动化用例执行结束时（无论成败）发送一条通知
2. **熔断触发**：自动化计划因熔断条件被暂停时发送一条通知

**验收标准：**
- AC1：钉钉消息卡片包含：触发场景（执行完成 / 熔断）、计划名、项目名、环境、总用例数、通过数、失败数、跳过数、执行时间、报告跳转链接
- AC2：手动计划不发送任何通知
- AC3：推送失败时记录日志，不阻塞计划状态变更
- AC4：同一事件不重复发送（如熔断后又被手动恢复执行完成，分别发送一次）

### 5.6 测试计划

#### FR-PLAN-001：创建测试计划

**字段：** 计划名称、计划类型（自动化/手动）、测试类型（API/E2E）、用例集、目标环境、通知渠道、失败重试次数（0-3，仅自动化）、熔断配置（仅自动化）

> **说明：** 测试计划为项目级资源，不绑定特定分支配置。创建计划时，从项目内任意分支的用例中勾选，不限制分支来源。

**验收标准：**
- AC1：自动化计划必填环境和通知渠道；手动计划环境和通知渠道可选
- AC2：用例集至少 1 条，最多 1000 条；可从项目内任意分支配置中选择用例，支持跨分支混选
- AC3：熔断配置默认值：连续失败 5 条暂停，失败率 50% 暂停
- AC4：草稿状态计划可编辑，执行中/已完成/已归档不可编辑

#### FR-PLAN-002：自动化计划执行

**验收标准：**
- AC1：点击「执行」后，平台筛选「已自动化」且非 Flaky 的用例逐条执行
- AC2：执行过程实时显示进度（已执行 N/总 M）
- AC3：单条用例失败按重试次数自动重试
- AC4：所有自动用例执行完成后，平台开放「待自动化」+「Flaky」用例的手动录入入口
- AC5：所有用例（自动+手动）完成后，计划状态变为「已完成」

#### FR-PLAN-003：手动计划执行

**验收标准：**
- AC1：点击「执行」后，所有用例直接进入手动录入模式
- AC2：每条用例展示：标题、模块、前置、步骤、预期
- AC3：录入项：结果（通过/失败）、备注（可选）、实际执行时长（可选）
- AC4：录入界面支持筛选：全部 / 待录入 / 已录入通过 / 已录入失败
- AC5：每条用例录入后自动保存为草稿（即时保存）
- AC6：所有用例录入完成后，用户手动点击「确认完成」按钮，计划状态变为「已完成」

#### FR-PLAN-007：用例处理人分配（Lead 分发协作）

**描述：** 测试 Lead 可在计划中为每条用例指定处理人，避免多人同时录入冲突。

**验收标准：**
- AC1：计划详情页支持「分配处理人」操作（单条 / 批量）
- AC2：处理人候选范围为该项目成员
- AC3：未分配处理人的用例所有项目成员均可录入
- AC4：已分配处理人的用例，仅该处理人可录入；其他人只读
- AC5：Lead（项目管理员或测试角色）可随时重新分配处理人
- AC6：处理人字段记录在执行结果中，导出报告时可见

#### FR-PLAN-008：自动化计划完成后的手动录入触发

**描述：** 自动化计划执行完成后，若计划包含「待自动化」或「Flaky」用例，开放手动录入入口。

**验收标准：**
- AC1：自动化执行完成后，**仅当存在待手动录入的用例时**，计划详情页显示「开始手动录入」按钮
- AC2：若计划全部为已自动化用例，自动执行完成后直接进入「已完成」状态，不显示手动录入按钮
- AC3：状态横幅明确提示："自动执行已完成，N 条用例待手动录入"
- AC4：手动录入完成后，需用户点击「确认完成」整个计划状态才变为「已完成」

#### FR-PLAN-009：计划重新打开

**描述：** 已完成的计划可重新打开，用于补充录入或重新执行。

**验收标准：**
- AC1：状态为「已完成」的计划，提供「重新打开」按钮
- AC2：点击后计划状态变为「执行中」，已有用例执行结果保留不变，可继续补充录入
- AC3：仅项目管理员和计划创建者有权操作
- AC4：操作记录到审计日志

#### FR-PLAN-004：执行队列

**验收标准：**
- AC1：同项目同测试类型的自动化计划不并发，排队执行（不同项目可并行）
- AC2：手动计划无队列限制
- AC3：队列状态可在计划列表查看（等待中 / 第 N 位）

#### FR-PLAN-005：熔断与暂停（仅自动化计划）

**验收标准：**
- AC1：触发熔断条件后，当前计划暂停
- AC2：已执行用例结果保留
- AC3：推送钉钉告警
- AC4：暂停的计划可手动「恢复」（继续执行剩余用例）或「终止」（整个计划标记完成，未执行用例标记为「未执行」）

#### FR-PLAN-006：计划归档

**验收标准：**
- AC1：「已完成」状态的计划可手动归档
- AC2：「已归档」计划只读，不可再执行
- AC3：归档后历史执行记录保留

> **状态机说明（两套独立）：**
>
> **自动化计划：**
> ```
> 草稿 → 执行中 → 已完成 → 已归档
>           ↓
>       已暂停（熔断）→ 可恢复继续执行 / 终止后变为已完成
> ```
>
> **手动计划（无暂停、无熔断）：**
> ```
> 草稿 → 执行中（保存即进入）→ 已完成（用户手动点击「完成」）→ 已归档
> ```
> - 手动计划的「执行中」是一个长期持续状态，未点击「完成」之前一直保持
> - 测试人员可分多次进入计划录入结果

### 5.7 执行引擎

#### FR-EXEC-001：脚本调用

**描述：** 平台调用本地测试运行器执行脚本，并解析结果。

**验收标准：**
- AC1：根据 script_ref.file 拼接完整脚本路径
- AC2：脚本文件不存在时，用例标记「脚本已移除-跳过」
- AC3：将全局变量与选定环境的环境变量合并后注入测试进程（环境变量优先级 > 全局变量，详见 FR-ENV-003）
- AC4：解析测试运行器输出的 JUnit XML，提取每条用例结果（通过/失败/错误）
- AC5：失败用例的错误信息和调用栈完整保存
- AC6：解析测试运行器附加输出的「步骤级日志」（pytest 通过 fixture / hook 输出 JSON），提取每个步骤的状态、耗时、API 请求响应、截图路径等

> **说明：** 执行引擎一期实现仅支持 pytest（含 pytest-playwright），未来扩展其他语言运行器作为单独 FR 处理。具体调用命令见架构文档。

#### FR-EXEC-003：步骤级执行日志（TEA 配合）

**描述：** TEA 生成的脚本需在执行时输出步骤级日志，平台据此渲染失败详情界面。

**TEA 脚本输出规范（每条用例额外输出一份 JSON）：**

```json
{
  "case_id": "auth_login_redirect_to_dashboard",
  "status": "failed",
  "duration_ms": 3200,
  "steps": [
    { "seq": 1, "action": "访问 /login 页面", "status": "passed", "duration_ms": 800 },
    { "seq": 2, "action": "输入正确邮箱和密码", "status": "passed", "duration_ms": 400 },
    { "seq": 3, "action": "点击「登录」按钮", "status": "failed", "duration_ms": 2000,
      "error_summary": "预期跳转到 /dashboard，实际停留在 /login",
      "screenshot": "screenshots/auth_login_redirect_to_dashboard_step3.png"
    }
  ],
  "api_requests": [
    { "method": "POST", "url": "/api/login", "status": 500, "duration_ms": 1203,
      "request_body": "...", "response_body": "..." }
  ],
  "raw_stack_trace": "..."
}
```

**验收标准：**
- AC1：步骤级日志输出到 `<test_artifacts>/<case_id>.json`
- AC2：平台执行完用例后读取该文件，存入数据库供报告使用
- AC3：缺少步骤级日志的脚本（旧脚本兼容）仅展示原始堆栈，不影响通过/失败判定

#### FR-EXEC-002：执行结果状态

| 状态 | 含义 |
|------|------|
| 通过 | 测试运行器报告 PASSED |
| 失败 | 测试运行器报告 FAILED |
| 错误 | 脚本异常或超时 |
| 跳过 | Flaky 标记 / 无 script_ref / 脚本不存在 |
| 重试中 | 失败后按配置重试 |

### 5.8 结果与报告

#### FR-REPORT-001：报告下钻视图

**验收标准：**
- AC1：整体视图显示总数、通过数、失败数、跳过数、手动录入数、通过率、执行时长
- AC2：模块视图按模块汇总通过率

- AC4：失败详情按下方 FR-REPORT-004 规范展示

#### FR-REPORT-004：失败详情三层展示规范

**描述：** 失败用例点击展开后按三层递进展示，避免一次性堆砌技术信息。

**第一层：失败摘要（默认展示）**
- 失败步骤序号 + 步骤描述（如"第 3 步 - 点击「登录」按钮"）
- 一句话失败原因（如"预期跳转到 /dashboard，实际停留在 /login"）

**第二层：执行证据（默认展开）**
- 步骤清单：每步的状态（✅/❌/⏭）+ 耗时
- E2E 失败截图（缩略图，可点击放大）
- API 请求详情：方法 + URL + 状态码 + 请求体 + 响应体 + 耗时

**第三层：原始堆栈（默认折叠）**
- pytest 完整错误堆栈
- 折叠按钮：「展开技术细节」

**验收标准：**
- AC1：自动失败用例严格按三层展示
- AC2：手动失败用例仅显示测试人员备注（无三层结构）
- AC3：缺少步骤级日志（旧脚本）的失败用例仅显示第三层堆栈
- AC4：截图懒加载，点击展开时才加载

#### FR-REPORT-002：HTML 报告导出

**验收标准：**
- AC1：导出按钮在计划详情页可用
- AC2：500 条用例导出 < 30 秒
- AC3：HTML 自包含（无外部依赖），可离线浏览
- AC4：包含整体、模块、用例三层数据 + 失败详情

#### FR-REPORT-003：Excel 报告导出

**验收标准：**
- AC1：导出按钮在计划详情页可用
- AC2：500 条用例导出 < 15 秒
- AC3：Excel 包含用例 ID、标题、模块、状态、错误摘要、执行时长

### 5.9 操作日志

#### FR-LOG-001：日志记录

**记录范围：**
- 用户：创建、修改、删除、绑定项目
- 项目：创建、修改配置
- 分支配置：创建、归档、恢复
- 用例管理：创建、修改、删除、导入、归档、Flaky 标记、更新脚本、更新用例

- 配置：环境、通知渠道变更

**验收标准：**
- AC1：每条日志含：操作人、时间（精确到秒）、操作类型、对象名称、变更摘要
- AC2：变更摘要以 JSON 形式记录前后值（敏感字段如 Webhook URL 不记录原值）
- AC3：日志保留 ≥ 3 年
- AC4：仅系统管理员可看到所有项目日志，其他角色仅看所属项目日志

#### FR-LOG-002：日志查询

**验收标准：**
- AC1：支持按操作人、时间范围、操作类型筛选
- AC2：支持关键字搜索（对象名称）

## 九、外部依赖

### 9.1 TEA 配合事项

TEA 需在生成测试脚本时，同步输出统一的结构化用例文件 `tea-cases.json`，**并要求脚本在执行时输出步骤级日志**。

**TEA 负责人需实现：**

1. 在 TA / ATDD workflow 中增加「生成用例清单」步骤
2. **一个测试场景 = 一个独立 `.py` 文件**：
   - 文件名采用描述式 slug（snake_case，以 `test_` 开头）
   - 文件名在所在目录内保证唯一，必要时追加简短后缀（如 `_v2`）
   - 共享 fixture / helper 放到 `conftest.py` 或 `support/` 目录
3. **单文件输出用例清单**：API 和 E2E 用例合并在同一个 `tea-cases.json` 中，通过每条用例的 `type` 字段（`api` / `e2e`）区分
4. **`id` 字段采用可读格式**：`{module}_{slug}`（如 `auth_login_redirect_to_dashboard`），其中：
   - `module` 为模块缩写码小写（如 `auth`）
   - `slug` 为脚本文件名去掉 `test_` 前缀和 `.py` 后缀的部分
   - **作为 TEA 端的可读 ID，便于人工识别和反查**
5. `script_ref.file` 为相对于项目根目录的脚本路径，用于执行引擎定位脚本
6. `script_ref.function` 与文件主函数同名，保留用于日志和兜底
7. **脚本执行时输出步骤级日志**（详见 FR-EXEC-003）：
   - 通过 pytest fixture / hook 在每个测试步骤执行后追加 JSON 日志
   - 日志写入 `<test_artifacts>/<case_id>.json`
   - 包含每步的状态、耗时、错误摘要、API 请求响应、截图路径
   - 此日志驱动平台「失败详情三层展示」（FR-REPORT-004）

### 9.2 用例字段规范

`tea-cases.json` 必须包含：`id`、`type`、`title`、`module`、`priority`、`preconditions`、`steps`、`expected_result`、`script_ref`，可选 `submodule`。

**示例：**

```json
{
  "generated_at": "2026-04-13T10:00:00Z",
  "test_cases": [
    {
      "id": "auth_login_redirect_to_dashboard",
      "type": "e2e",
      "title": "登录后跳转首页",
      "module": "AUTH",
      "submodule": "LOGIN",
      "priority": "P0",
      "preconditions": ["用户账号已存在", "处于未登录状态"],
      "steps": [
        { "seq": 1, "action": "访问 /login 页面" },
        { "seq": 2, "action": "输入正确邮箱和密码" },
        { "seq": 3, "action": "点击「登录」按钮" }
      ],
      "expected_result": "页面跳转到 /dashboard，显示欢迎消息",
      "script_ref": {
        "file": "tests/e2e/auth/test_login_redirect_to_dashboard.py",
        "function": "test_login_redirect_to_dashboard"
      }
    },
    {
      "id": "auth_login_missing_password",
      "type": "api",
      "title": "登录接口缺少密码字段",
      "module": "AUTH",
      "submodule": "LOGIN",
      "priority": "P1",
      "preconditions": ["接口服务可访问"],
      "steps": [
        { "seq": 1, "action": "POST /api/login，缺少 password 字段" }
      ],
      "expected_result": "返回 400，错误码 PARAM_MISSING",
      "script_ref": {
        "file": "tests/api/auth/test_login_missing_password.py",
        "function": "test_login_missing_password"
      }
    }
  ]
}
```
