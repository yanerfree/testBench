# TestForge — AI 测试用例与脚本生成

从 testBench 平台生成的 task JSON 中读取接口信息和业务规则，系统化应用 6 种测试设计技法生成高质量测试用例，自动导入 testBench 平台，生成可执行的 pytest+httpx 脚本。

## 核心原则

- **断言对齐预期** —— assert 必须验证预期结果中描述的可观测点（状态码、字段值、错误提示），不准只断 200
- **数据随机化** —— 可重复输入用 `uuid.uuid4().hex[:8]` 生成随机后缀；环境相关值走 `os.getenv` 带默认值
- **依赖资源动态构造** —— 不假设数据库已有特定数据；前置数据走 fixture 创建，用完即清理
- **脚本自包含** —— 生成的脚本不依赖被测项目仓库的代码，所有 fixture 在脚本内自行定义
- **遵循 testBench 规范** —— 文件命名 `test_{slug}.py`、类命名 `Test{PascalCase}`、遵循 `project-context.md` 中的 TEA 约束
- **质量门禁** —— 用例设计完成后自评分（完整性/准确性/有效性/可执行性），总分 ≥75 才继续

## 输入

task JSON，来源二选一：
1. 读取 `testforge/tasks/` 目录中最新的 `status: pending` 文件
2. 用户直接在对话中粘贴 JSON

task JSON 必含字段：

| 字段 | 说明 |
|------|------|
| `platform.api_url` | testBench API 地址 |
| `platform.token` | JWT 认证 token |
| `platform.project_id` | 目标项目 UUID |
| `platform.branch_id` | 目标分支 UUID |
| `target.module` | 目标模块名（如 rfc-connector） |
| `target.submodule` | 子模块名（可选） |
| `target.script_dir` | 脚本输出目录（如 tests/api/rfc_connector） |
| `interface.name` | 接口/模块名称 |
| `interface.base_url` | 被测系统地址 |
| `interface.auth` | 认证方式（JWT / API Key / 无） |
| `interface.endpoints[]` | 接口列表（method、path、headers、body、response） |
| `business_rules[]` | 业务规则列表（字符串数组） |

## 执行流程

### Step 1：读取并解析 Task

1. 检查 `testforge/tasks/` 目录，找最新的 pending task
2. 如果没有 pending task，提示用户粘贴 task JSON 或在平台上创建
3. 解析 JSON，提取接口信息、业务规则、平台认证信息
4. 验证 token 有效性：`GET {api_url}/api/auth/me`，带 `Authorization: Bearer {token}`
5. 如果 token 无效，提示用户在平台上重新生成 task

### Step 2：加载知识库

读取以下文件作为测试设计的参考依据：

**测试技法**（`testforge/knowledge/techniques/`）：
- equivalence-partitioning.md — 等价类划分
- boundary-value-analysis.md — 边界值分析
- decision-table.md — 决策表
- state-transition.md — 状态迁移
- scenario-method.md — 场景法
- error-guessing.md — 错误推测

**通用模式**（`testforge/knowledge/patterns/`）：
- common-edge-cases.md — 通用边界用例
- auth-patterns.md — 认证授权模式
- crud-patterns.md — CRUD 模式
- validation-patterns.md — 输入校验模式

**领域知识**（`testforge/knowledge/domain/`）：
- 如果存在与当前接口相关的领域文件，一并加载

**历史反馈**（`testforge/feedback/`）：
- 如果存在历史反馈文件，提取其中的 bad/missing 记录，避免重复相同错误

### Step 3：设计测试用例

对 task 中的每个 endpoint + 每条 business_rule，系统化应用 6 种技法：

**3.1 等价类划分**
- 对每个请求参数，划分有效等价类和无效等价类
- 每个无效等价类生成一条用例

**3.2 边界值分析**
- 对有 max_length、range、pattern 的字段，生成边界值用例
- 必须包含：恰好在边界上、刚好越界

**3.3 决策表**
- 对有多条件组合的场景（如 connectType × openType × 连通测试结果），构建决策表
- 每条规则生成一条用例

**3.4 状态迁移**
- 识别操作之间的依赖关系（如"必须先连通测试才能保存"）
- 生成合法转换和非法转换用例

**3.5 场景法**
- 生成端到端基本流：创建 → 查询 → 编辑 → 删除
- 生成备选流：每个步骤的异常分支

**3.6 错误推测**
- 对照 common-edge-cases.md 和 error-guessing.md
- 补充安全测试（注入）、并发、幂等等用例

### Step 4：质量门禁评分

对生成的用例集进行自评：

| 维度 | 满分 | 检查项 |
|------|------|--------|
| 完整性 | 25 | 所有 endpoint 和 business_rule 是否都被覆盖 |
| 准确性 | 25 | 步骤是否具体可执行、预期结果是否可量化验证 |
| 有效性 | 25 | 用例是否能发现真实 bug（不是废话用例） |
| 可执行性 | 25 | 前置条件是否可构造、测试数据是否有来源 |

**红线规则**：
- 预期结果中出现"正确"、"成功"、"正常"等模糊词 → 替换为具体的状态码、字段值、错误提示
- 测试步骤中出现硬编码值（如 `zhangsan`、`admin123`） → 替换为 `${变量}` 引用
- 缺少前置条件 → 补充

总分 < 75 → 自动修正后重新评分（最多 2 轮）

### Step 5：展示用例并等用户确认

以 Markdown 表格格式展示所有用例：

```markdown
## {接口名} 测试用例

### {场景分组}

| 编号 | 用例名称 | 前置条件 | 测试步骤 | 预期结果 | 类型 | 设计技法 |
|------|---------|---------|---------|---------|------|---------|
| TC-001 | ... | ... | ... | ... | API | 等价类 |
```

等待用户确认。用户可以：
- 确认 → 继续
- 修改 → 根据用户修改调整后再确认
- 补充 → 新增用例后再确认

### Step 6：导入用例到 testBench 平台

将确认后的用例转为 testBench 导入格式：

```json
[
  {
    "tea_id": "tf_{module}_{slug}",
    "title": "{用例名称}",
    "type": "api",
    "module": "{target.module}",
    "submodule": "{target.submodule}",
    "priority": "{P0|P1|P2|P3}",
    "preconditions": "{前置条件}",
    "steps": [{"action": "{测试步骤}"}],
    "expected_result": "{预期结果}",
    "script_ref": {
      "file": "{target.script_dir}/test_{slug}.py",
      "class": "Test{PascalCase}",
      "func": "Test{PascalCase}::test_{specific}"
    }
  }
]
```

**tea_id 规则**：`tf_{module}_{slug}`，其中 slug 从用例名称生成（小写、下划线分隔、去特殊字符）。

将 JSON 保存为临时文件，然后调用 testBench 导入 API：

```bash
curl -X POST "{api_url}/api/projects/{project_id}/branches/{branch_id}/cases/import" \
  -H "Authorization: Bearer {token}" \
  -F "file=@/tmp/tf-cases-import.json"
```

检查响应：确认 new + updated 数量，报告 skipped 及原因。

### Step 7：生成 pytest 脚本

对每条 type=API 的用例，生成 pytest+httpx 脚本。

**文件结构**（遵循 testBench project-context.md）：

```python
"""
API 测试: {用例标题}
tea_id: tf_{module}_{slug}
Generated by TestForge
"""
from __future__ import annotations
import os
import uuid

import httpx
import pytest

BASE_URL = os.getenv("BASE_URL", "{interface.base_url}")
RAND = uuid.uuid4().hex[:8]


@pytest.fixture(scope="module")
def client():
    with httpx.Client(base_url=BASE_URL, timeout=30) as c:
        yield c


@pytest.fixture(scope="module")
def token(client):
    resp = client.post(
        "{auth.login_endpoint}",
        json={auth.credentials},
    )
    assert resp.status_code == 200
    return resp.json()["data"]["access_token"]


@pytest.fixture(scope="module")
def headers(token):
    return {
        "Authorization": token,
        "Content-Type": "application/json; charset=UTF-8",
    }


# 依赖资源 fixture（如需要）
@pytest.fixture(scope="module")
def dep_{resource}(client, headers):
    # 创建前置数据
    resp = client.post("...", json={...}, headers=headers)
    assert resp.status_code in (200, 201)
    resource_id = resp.json()["data"]["id"]
    yield resource_id
    # 清理
    client.delete(f".../{resource_id}", headers=headers)


class Test{PascalCase}:
    """对应用例: TC-{xxx} {用例标题}"""

    def test_{specific}(self, client, headers):
        resp = client.{method}(
            "{path}",
            json={request_body},
            headers=headers,
        )
        # 精确断言 — 对齐用例预期结果
        assert resp.status_code == {expected_status}
        data = resp.json()
        assert data["{field}"] == {expected_value}
```

**脚本输出位置**：`{target.script_dir}/test_{slug}.py`

如果目标目录不存在，自动创建：`mkdir -p {target.script_dir}`

### Step 8：更新 tea-cases.json

读取项目根目录的 `tea-cases.json`，在 `cases` 数组中追加新条目：

```json
{
  "tea_id": "tf_{module}_{slug}",
  "title": "{用例标题}",
  "module": "{target.module}",
  "submodule": "{target.submodule}",
  "type": "api",
  "level": "api",
  "priority": "{priority}",
  "script_ref": {
    "file": "{target.script_dir}/test_{slug}.py",
    "class": "Test{PascalCase}",
    "func": "test_{specific}"
  },
  "tags": ["testforge", "ai-generated"]
}
```

更新 `summary` 中的统计数字。

### Step 9：可选执行 + 自愈

用户确认后，可以选择执行脚本：

```bash
cd {project_root} && python -m pytest {target.script_dir}/test_{slug}.py -v --tb=long
```

**如果全部通过**：输出成功报告。

**如果有失败**，进入自愈循环（最多 3 轮）：

1. 解析 pytest 输出，定位失败的测试和断言
2. 诊断根因：

| 错误类型 | 根因 | 修复方向 |
|---------|------|---------|
| ConnectionError | 被测服务不可达 | 检查 BASE_URL，**不修改脚本** |
| 401 Unauthorized | token 无效/过期 | 重新登录获取 token |
| 404 Not Found | URL 路径错误 | 检查 endpoint path |
| 400 Bad Request | 请求参数格式错误 | 修改请求 body |
| AssertionError: status_code | 预期状态码与实际不符 | 检查接口实际行为，**询问用户** |
| AssertionError: field value | 响应结构与预期不符 | 调整断言匹配实际结构 |
| KeyError | 响应 JSON 结构不同 | 修改字段访问路径 |

3. 修改脚本，重新执行
4. **绝不放宽断言来通过测试** — 如果实际行为和预期不符，停下来问用户："接口实际返回 X，但用例预期是 Y，以哪个为准？"

### Step 10：更新 Task 状态 + 输出报告

如果 task 来自文件（`testforge/tasks/`），更新其 `status` 为 `completed`。

输出最终报告：

```
✅ TestForge 生成完成

| 项目 | 数量 |
|------|------|
| 生成用例 | {N} 条 |
| 导入平台 | 新增 {X} 条，更新 {Y} 条 |
| 生成脚本 | {M} 个文件 → {target.script_dir}/ |
| tea-cases.json | 新增 {M} 条 |
| 执行结果 | {passed}/{total} 通过（自愈 {rounds} 轮） |

用例设计技法分布：
- 等价类: {n1} 条
- 边界值: {n2} 条
- 决策表: {n3} 条
- 状态迁移: {n4} 条
- 场景法: {n5} 条
- 错误推测: {n6} 条
```
