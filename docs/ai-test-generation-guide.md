# AI 测试落地指南 — 从接入到质量保证

## 要解决的真实问题

测试团队面对 AI 测试工具时，真正的困难不是"怎么生成脚本"，而是：

1. **测试人员不会写代码**，生成的 pytest 脚本看不懂、改不了、不敢用
2. **生成的测试不可信**，不知道它到底验证了什么，通过了也不敢说没 bug
3. **维护成本不可控**，接口一改，一堆脚本挂了，修起来比手动写还累
4. **没有衡量标准**，无法回答"AI 测试到底帮了多少忙"

本文不讲工具能力，讲**怎么落地**。

---

## 一、接入路径 — 三种角色、三条路

### 路径 A：测试人员（不写代码）

**目标**：不碰代码，通过平台操作完成 AI 测试的接入和执行。

```
第 1 天：在平台上创建项目 → 配置 Git 仓库 → 同步分支
         → 导入 tea-cases.json → 在用例管理中看到生成好的用例
         → 检查每条用例的步骤、接口、参数是否合理

第 2 天：在平台上配置测试环境（base_url / 账号密码）
         → 创建测试计划 → 选择用例 → 执行
         → 在报告中查看步骤级结果

日常：  → 每次 Git 同步自动更新用例
         → 关注执行报告中的失败步骤
         → 对可疑用例标记 Flaky 或提 bug
```

**这条路径要求开发/AI 已经提前生成了 tea-cases.json 和脚本。** 测试人员的职责是审核用例质量、配置环境、执行并判读结果 — 不需要写代码。

**当前缺失**：
- 平台没有"用例审核"状态（待审核 → 已审核 → 已拒绝）
- 没有"AI 生成置信度"评分，测试人员不知道哪些用例需要重点审
- 没有批量执行后的趋势对比（本次 vs 上次，哪些用例从通过变失败）

### 路径 B：测试工程师（会写代码）

**目标**：用 AI 生成初稿，人工审核修改后纳入正式用例库。

```
Step 1：AI 扫描接口生成 test_*.py 脚本（初稿）
Step 2：人工 code review 每个脚本 ← 这步不可省略
        ✓ 断言是否验证了业务逻辑，而不只是 status_code == 200
        ✓ 测试数据是否有意义（不是随机字符串）
        ✓ 前置数据清理是否完整（不会污染下次执行）
        ✓ 边界值是否覆盖到
Step 3：修改后 commit 到脚本仓库
Step 4：同步到平台 → 正式纳入回归
```

**关键原则**：AI 生成是"第一版草稿"，不是"最终交付"。人工审核是质量门禁。

**当前缺失**：
- 没有 code review 工作流（谁审、审什么、审完怎么标记）
- 脚本只能查看不能在平台上编辑（需要回到 IDE）

### 路径 C：开发人员（写接口的人顺手生成测试）

**目标**：开发完接口后，一条命令生成测试，作为接口的"活文档"。

```
# 开发完 users 模块后
/generate-api-tests users

# 自动产出：
# - tests/api/users/test_create_user_success.py
# - tests/api/users/test_create_user_duplicate.py
# - tests/api/users/test_list_users.py
# - tea-cases.json 更新
# - pytest --collect-only 验证通过

# commit + push → 平台自动同步
```

**对开发的要求**：
- Schema 写 Field 约束（这不是为了测试，是 API 本身的质量要求）
- 路由用声明式权限（`Depends(require_role(...))`）
- 这两件事开发本来就该做，做到位了 AI 才能生成有效的测试

## 二、质量保证 — 怎么知道 AI 生成的测试有没有用

### 2.1 生成质量评估（生成时）

AI 生成的测试常见的三个问题：

| 问题 | 表现 | 检测方法 |
|------|------|---------|
| **弱断言** | `assert resp.status_code == 200` 就结束了，没验证返回数据 | 扫描脚本，统计每个 test 函数的 assert 数量，<2 个标记为"弱覆盖" |
| **假阳性** | 测试永远通过，因为根本没调到真实逻辑 | 跑一遍"故意破坏"测试：把被测接口返回值改错，看测试是否能检出 |
| **数据耦合** | 测试依赖数据库里的特定数据，换个环境就挂 | 检查脚本是否在 setup 阶段自己创建测试数据，而不是依赖已有数据 |

**建议引入的机制**：

```
生成脚本后自动评分（0-100）：
  - 断言丰富度：每个 test 有 ≥3 个 assert → +30 分
  - 数据自治：setup 中创建了前置数据 → +20 分
  - 异常覆盖：有 4xx 状态码的测试场景 → +20 分
  - 清理完整：测试结束后清理了创建的数据 → +15 分
  - 步骤分层：用了 tea_step 且有 setup/action/verify 三阶段 → +15 分

评分 < 60 → 标记为"需人工审核"
评分 ≥ 80 → 可直接纳入回归
```

### 2.2 执行质量度量（运行时）

单纯看 pass/fail 数量没有意义。要回答的问题是：**这些测试发现了几个真 bug？**

| 指标 | 含义 | 计算方式 | 健康范围 |
|------|------|---------|---------|
| **有效失败率** | 失败的测试中，多少是真 bug | 真 bug 数 / 总失败数 | > 70% |
| **误报率** | 失败中有多少是测试自身问题 | 环境/数据/脚本问题导致的失败 / 总失败数 | < 30% |
| **漏检率** | 线上 bug 中有多少是测试没覆盖的 | 线上 bug 且无对应用例 / 总线上 bug 数 | < 20% |
| **Flaky 率** | 同样的代码，测试结果不稳定 | Flaky 用例数 / 总用例数 | < 5% |
| **维护成本** | 接口变更后需要修改多少脚本 | 每次 API 变更影响的脚本数 | 越低越好 |

**当前缺失**：
- 平台没有"有效失败 vs 误报"的标记功能
- 没有 Flaky 趋势统计
- 没有"测试发现 bug"的闭环追踪

### 2.3 覆盖差距分析（持续）

不是"有多少用例"，而是"还差什么"。

```
API 覆盖差距分析：
  ┌──────────────────┬────────┬────────┬────────┐
  │ 端点              │ 正向   │ 权限   │ 边界值 │
  ├──────────────────┼────────┼────────┼────────┤
  │ POST /api/users  │   ✅   │   ✅   │   ❌   │ ← 缺少 username 长度边界测试
  │ GET /api/users   │   ✅   │   ❌   │   ✅   │ ← 缺少无权限访问测试
  │ PUT /api/users/x │   ❌   │   ❌   │   ❌   │ ← 完全没有测试
  │ DELETE /api/users│   ✅   │   ✅   │   -    │
  └──────────────────┴────────┴────────┴────────┘

  总覆盖率: 7/11 (64%)
  建议优先补充: PUT /api/users/{id} 的全部场景
```

**实现方式**：扫描所有路由端点 → 对比 tea-cases.json 中的 apiEndpoint → 输出差距矩阵。

## 三、开发侧真正需要做的事（不是写文档，是写好代码）

与其要求开发"额外产出标准文件"，不如把要求嵌入到编码规范中 — 开发做到这几点，代码本身就是 AI 测试的最佳输入：

### 3.1 Schema 即契约

```python
# ❌ 没用的写法 — AI 只知道有个 string 字段
class CreateUserRequest(BaseSchema):
    username: str
    password: str

# ✅ 有用的写法 — AI 能推导出 6 个测试场景
class CreateUserRequest(BaseSchema):
    username: str = Field(min_length=3, max_length=50, pattern=r"^[a-zA-Z0-9_]+$")
    password: str = Field(min_length=8, max_length=128)
    email: str = Field(max_length=200)
    role: Literal["admin", "developer", "tester", "guest"] = "tester"
```

**这不是为了 AI 而额外做的事，这是 API 开发的基本质量要求。** Field 约束就是输入校验，没有它 API 本身就有安全风险。

### 3.2 错误码即测试用例

```python
# ❌ 模糊的错误处理
if existing_user:
    raise HTTPException(status_code=400, detail="error")

# ✅ 明确的错误码 — AI 能直接生成对应场景
if existing_user:
    raise AppError(code="USER_DUPLICATE", message="用户名已存在", status_code=409)
```

每个 `AppError(code=...)` 就是一个异常测试场景。错误码规范 → 异常用例自动推导。

### 3.3 业务规则写在注释或 docstring 里

```python
@router.post("/api/projects/{pid}/plans")
async def create_plan(...):
    """创建测试计划。
    
    业务规则:
    - 同项目下计划名不可重复
    - 归档状态的项目不允许创建计划
    - 计划类型: smoke / regression / full
    """
```

AI 扫描 docstring 就能推导出重复名称、归档项目、非法类型三个异常场景。不需要单独写需求文档。

## 四、落地节奏 — 分三步走

### 第一步：跑通一个模块（1-2 周）

选一个接口数量 5-10 个的模块（如 auth 或 users），完整走一遍：

```
AI 生成脚本 → 人工审核 → 修改 → 入库 → 执行 → 查看报告
```

**验收标准**：
- 生成的脚本无需大幅修改即可执行通过
- 报告中的步骤描述能让不看代码的人理解"测了什么"
- 至少发现 1 个真实 bug 或遗漏

### 第二步：建立审核机制（2-4 周）

- 定义 AI 生成脚本的审核 checklist
- 在平台上标记用例状态：AI生成 → 待审核 → 已审核 → 纳入回归
- 统计首次生成通过率、审核修改率
- 识别 AI 生成的常见问题，反馈到 Skill 模板中改进

### 第三步：度量驱动（持续）

- 接入覆盖差距分析，可视化"哪些端点没测到"
- 跟踪有效失败率和误报率
- 对比 AI 生成用例 vs 人工用例的 bug 发现能力
- 根据数据决定哪些模块适合全量 AI 生成，哪些需要人工补充

## 五、testBench 平台需要补的能力

| 能力 | 优先级 | 说明 |
|------|--------|------|
| **用例审核状态** | P0 | AI生成 → 待审核 → 已审核，区分"可信"和"待验证"的用例 |
| **生成质量评分** | P0 | 断言丰富度、数据自治、异常覆盖 → 自动打分 |
| **覆盖差距矩阵** | P1 | 扫描路由 vs 已有用例 → 显示哪些端点/场景没覆盖 |
| **执行趋势对比** | P1 | 本次 vs 上次执行，哪些用例状态变化了 |
| **Flaky 趋势** | P1 | 同一用例最近 N 次执行的通过率 |
| **脚本在线编辑** | P2 | 测试人员能直接在平台上修改脚本，不用回 IDE |
| **失败分类标记** | P2 | 失败原因：真 bug / 环境问题 / 脚本问题 / 数据问题 |
| **Swagger 驱动生成** | P2 | 直接上传 Swagger 文件，一键生成全量测试 |

---

## 六、一句话总结

**AI 测试的价值不在于"生成了多少脚本"，而在于"发现了多少真 bug、减少了多少人工重复劳动"。** 落地的关键是：选一个模块跑通全链路、建立人工审核门禁、用度量数据证明效果。
