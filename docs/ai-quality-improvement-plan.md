# 工作质量改善方案（通用规范）

适用于 testBench 项目所有功能开发，不限于某个特定模块。

## 问题复盘

| 问题现象 | 根因 |
|---------|------|
| 改了代码不测试 | 只看编译通过就提交，没有从用户角度验证 |
| UI 难看改不好 | 没有先看项目中好的页面学习标准，闭门造车 |
| 功能做一半 | 只做了 happy path，没考虑用户实际使用场景 |
| 同一问题反复修 | 修一个破一个，缺少回归验证 |
| 用户说了才改 | 不主动思考，等用户指出才行动 |
| 过度复杂设计 | 加了用户不需要的按钮/选项/步骤 |

**核心问题：没有站在用户角度做事，只在完成代码层面的任务。**

---

## 执行规则

### 1. 每次改完必须自测

适用所有场景：

| 变更类型 | 自测方法 |
|---------|---------|
| UI/页面 | Playwright 截图验证实际渲染效果 |
| 功能 | 走完整流程：增→查→改→删 |
| AI/生成 | 实际生成一份结果，检查内容质量 |
| API | curl 或脚本调用，验证请求响应 |
| 表单 | 空提交校验、边界值、长文本 |

### 2. 做之前先问先看

- 改 UI → 先截图当前状态 + 看项目中好的页面做参照
- 改功能 → 不确定就问用户，不要猜
- 改 prompt → 先读参考文档，提取具体标准
- 不自作主张删/加功能

### 3. 一次一件事，做完做好

- 一个功能改完 → 自测通过 → 提交 → 再做下一个
- 不要同时开多个半成品

### 4. 默认最简

- 用户要一个按钮就给一个，不要给两个
- 不加用户没要求的选项/步骤/弹窗
- 改之前确认，加之前确认

### 5. 提交前心理检查

1. 如果我是用户，打开这个页面，我满意吗？
2. 我有没有实际看过改完的效果？还是只看了代码？
3. 用户上次吐槽的问题，这次真的解决了吗？
4. 有没有改出新问题？

---

## 提交规范

涉及 UI 或功能变更的 commit，必须包含验证结果：

```
feat/fix: 描述

验证：
- 截图确认 xxx 正常
- 完整流程测试通过
- 边界情况正常
```
