# 用例质量评审 Skill

## Step 1 — 收集用例

1. 调用 `tb_list_cases` 获取目标模块的所有用例
2. 调用 `tb_list_api_tree` 获取对应的 API 接口定义
3. 统计用例数量、优先级分布、类型分布

## Step 2 — 四维度评审

从以下 4 个维度逐一评审：

### 完整性（30%）
- 所有 API 接口都有对应用例吗？
- 每个接口有正向 + 异常场景吗？
- 关键业务路径有端到端用例吗？
- 安全测试覆盖了吗？

### 准确性（25%）
- 预期结果是否具体可验证（不是"功能正常"这类笼统描述）？
- 每步是否只有一个操作？
- 测试数据是否具体而非笼统？

### 有效性（25%）
- 每条用例能发现至少一类缺陷吗？
- 是否有逻辑重复的用例？
- 优先级分配是否合理（P0 ≤ 15%）？

### 可执行性（20%）
- 前置条件是否独立（不依赖其他用例的执行结果）？
- 步骤是否足够详细，新人能执行？
- 环境变量引用是否正确？

## Step 3 — 输出评审报告

输出 JSON 格式的评审报告：
- 总分（0-100）
- 各维度得分
- 问题列表（每条标注维度、严重程度、具体用例）
- 覆盖矩阵（接口 vs 用例）
- 改进建议

## 评分标准

- 90-100：优秀，可直接使用
- 75-89：良好，小幅修改后可用
- 60-74：一般，需要回炉补充
- 60 以下：不合格，建议重新设计
