# 用例生成 Skill v2

## 核心原则：页面操作视角

生成的是**手工功能测试用例**，面向测试人员在浏览器/客户端中操作。

- **所有步骤必须是页面操作**：打开页面、点击按钮、填写输入框、选择下拉项、查看列表、确认弹窗
- **禁止接口调用风格**：不写 `POST /api/...`、HTTP 状态码、curl、JSON body
- **预期结果必须是 UI 可见的**：Toast 提示内容、页面跳转、列表变化、表单校验红字、按钮状态

### 写作规范

- **元素锚点**：用可见文字/占位符/标签描述元素（如"点击文本为「保存」的按钮"），不写 CSS 选择器
- **数据随机化**：输入值用 `${变量名}` 引用前置条件中声明的变量，前置条件给出"生成规则 + 本次示例值"
- **可验证关键词**：预期必须含 `Toast 内容为` / `页面显示` / `列表第 N 行` / `跳转到` / `个数为` / `处于禁用状态` / `消失` / `颜色为`
- **一步一动作**：`输入 X 并点击保存` 必须拆为两步
- **步骤数 ≤ 8**：超过则拆分为多条用例

## 正确示例

```json
{
  "title": "创建服务 — 必填字段为空时校验拦截",
  "module": "服务管理/创建服务",
  "type": "api",
  "priority": "P1",
  "preconditions": "1. 已登录管理后台，拥有服务管理权限\n2. ${serviceName} 为随机服务名（生成规则：'svc_' + 8位随机字母；示例值：svc_kqzmwhxa）",
  "steps": [
    {"seq": 1, "action": "点击侧边栏「服务管理」菜单", "expected": "页面跳转到服务列表页"},
    {"seq": 2, "action": "点击右上角「创建服务」按钮", "expected": "弹出创建服务表单弹窗"},
    {"seq": 3, "action": "不填写任何字段，直接点击「确认」按钮", "expected": "「服务名称」输入框下方显示红色校验提示'请输入服务名称'；「服务类型」下拉框下方显示红色校验提示'请选择服务类型'"},
    {"seq": 4, "action": "在「服务名称」输入框中输入 ${serviceName}，其他必填字段仍留空", "expected": "「服务名称」的校验提示消失，「服务类型」的校验提示仍在"}
  ],
  "expected_result": "必填字段为空时表单校验阻止提交，逐个填写后校验提示逐个消失"
}
```

## 错误示例（禁止）

```json
{
  "title": "创建服务 — 必填字段缺失校验",
  "steps": [
    {"action": "POST /api/v1/services，body 不传 name 字段", "expected": "返回 422"},
    {"action": "POST /api/v1/services，body 不传 type 字段", "expected": "返回 422"}
  ]
}
```
问题：用接口调用代替了页面操作，预期只写 HTTP 状态码，没有 seq，没有前置条件。

## 执行流程

### Step 1：上下文收集

1. 调用 `tb_list_api_tree` 获取项目 API 接口列表（用于**理解业务逻辑**，不作为用例步骤）
2. 调用 `tb_get_api_node` 获取目标接口详情（字段定义、校验规则、枚举值）
3. 调用 `tb_list_cases` 查询同模块已有用例（避免重复）
4. 调用 `tb_get_folder_tree` 了解用例目录结构
5. 如果用户提供了项目文档路径，用 `Read` 读取提取功能点清单

### Step 2：维度规划

按四层优先级规划用例：

| 层级 | 说明 | 优先级 | 占比 |
|------|------|--------|------|
| 冒烟 | 核心功能正向路径（Happy Path） | P0 | ≤ 15% |
| 功能验证 | 各功能点完整正向验证 | P1 | 40-50% |
| 异常 & 边界 | 校验、空值、越界、权限、状态冲突 | P2 | 30-40% |
| 兼容 & 回归 | 跨模块影响、数据兼容、升级回归 | P3 | 5-10% |

维度清单：
- 正向路径（CRUD 完整流程）
- 表单校验（必填/格式/长度/枚举）
- 边界值（最小/最大/空/超长/特殊字符）
- 业务规则（唯一性/状态机/关联约束）
- 权限控制（未登录/无权限/跨租户）
- 异常处理（网络错误/并发冲突/数据不一致）
- 安全（XSS 注入/敏感信息泄露）
- UI 交互（Loading 状态/分页/排序/搜索）

输出规划的维度列表和每个维度预计生成的用例数。

### Step 3：逐维度生成

按维度依次生成用例。**每条用例必须包含**：

| 字段 | 必填 | 说明 |
|------|------|------|
| title | ✅ | `动作 — 场景细节`，如"创建服务 — 名称重复时提示已存在" |
| module | ✅ | 所属模块路径，如"服务管理/创建服务" |
| type | ✅ | `api` |
| priority | ✅ | P0/P1/P2/P3，严格按 Step 2 的占比 |
| preconditions | ✅ | 前置条件，包含环境准备和数据准备。每个 ${变量} 必须声明生成规则 + 示例值 |
| steps | ✅ | 步骤数组，每步 {seq, action, expected}。seq 从 1 开始连续编号 |
| expected_result | ✅ | 最终整体预期 |

**硬约束**：
- 每条用例只验证**一个测试点**，不混合多个场景
- 步骤中不出现接口路径/HTTP 方法/状态码/JSON
- 预期结果不允许模糊词：~~操作成功~~ / ~~显示正常~~ / ~~无报错~~ / ~~符合预期~~ / ~~正确显示~~ / ~~成功返回~~
- steps 的 seq 必须从 1 开始连续编号
- module 不能为空
- 和已有用例去重（标题或步骤场景相同的跳过）

### Step 4：自检

生成完毕后，对结果做自检：

1. **P0 比例检查**：P0 数量 / 总数 ≤ 15%，超标则降级最不关键的 P0 为 P1
2. **模糊词扫描**：扫描所有 expected，发现模糊词立即改写为具体描述
3. **字段完整性**：每条用例必须有 module、preconditions、seq 编号
4. **粒度检查**：标题含 `/` 或 `和` 且步骤验证多个点 → 拆分为独立用例
5. **维度覆盖**：对照 Step 2 规划的维度，缺少的补充 1-3 条

### Step 5：入库

将自检通过的用例逐条调用 `tb_create_case` 入库。

输出最终统计：
```
生成 N 条（P0: x, P1: y, P2: z, P3: w）
跳过 M 条重复
P0 占比: x/N = xx%（≤15% ✓）
```

## 质量红线（违反任何一条即不合格）

1. 步骤必须是页面操作，禁止接口调用风格
2. 预期必须含可验证关键词，禁止模糊词
3. 每条用例只验证一个测试点
4. P0 占比 ≤ 15%
5. 每条用例必须有 module、preconditions、steps.seq
6. 数据用 ${变量名} 引用，前置条件声明生成规则
7. 不生成和已有用例重复的内容
